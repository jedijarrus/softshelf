Schnellstart

Die häufigsten Aktionen in 1-2 Klicks:

Chocolatey-Paket freigeben

Tab Pakete → + Paket hinzufügen → Aus Chocolatey → Name suchen → + klicken. Fertig.

Winget-Paket freigeben

Tab Pakete → + Paket hinzufügen → Aus Winget → Name suchen (lokaler Microsoft-Catalog-Mirror) → + klicken. Im selben Panel zeigt der aufklappbare Block „In der Flotte gefunden" alle bereits installierten winget-Pakete die noch nicht whitelisted sind — auch von dort one-click aktivierbar.

Eigene MSI hochladen

Tab Pakete → + Paket hinzufügen → Eigene Datei → Datei wählen → Anzeigename checken → Hochladen. MSI-Metadaten werden automatisch extrahiert.

Software auf einem Client einsehen

Tab Kiosk-Clients → Klick auf eine Zeile öffnet die Agent-Detail-Page mit der gesamten installierten Software des Geräts (Tactical-Scan + winget-Inventur), Status-Badges (verwaltet/unverwaltet, Update verfügbar) und per-Row Aktionen (Updaten, Deinstallieren, Aktivieren, Entfernen).

Custom-Paket bearbeiten

Auf der Karte erscheint bei custom-Paketen ein Stift-Icon neben dem ×. Klick öffnet das Edit-Panel mit allen Feldern.

Neue softshelf-setup.exe bauen

Tab Einstellungen → ganz unten EXEs bauen. Dauert ~4 Minuten. Danach unter /download/softshelf-setup.exe verfügbar.

Client auf PC ausrollen

Tactical-Script deploy_via_trmm.ps1 auf den Agent anwenden. Oder manuell: softshelf-setup.exe herunterladen und als Admin ausführen.

Einstellung ändern

Tab Einstellungen → Feld editieren → Speichern. Gilt sofort, ohne Restart.

Wie "Installiert"-Status erkannt wird

Je nach Paket-Typ kommen die Daten aus zwei verschiedenen Quellen die kombiniert werden:

  • Choco-Pakete: Der Choco-Name (firefox) wird per bidirektionalem Substring-Match gegen die Windows-Anzeigenamen aus Tacticals Software-Scan geprüft (Mozilla Firefox (x64 en-US)). Zeigt das Kiosk-UI als „Installiert".
  • Custom-Pakete: Der beim Upload gespeicherte Detection-Name wird gegen die Windows-Anzeigenamen geprüft. Bei MSIs ist das der ProductName aus der MSI-Metadatenzone — stimmt fast immer ohne manuellen Eingriff. Bei EXEs muss der Admin den Detection-Name manuell eintragen (erscheint in „Apps & Features" auf dem Windows-PC).
  • Winget-Pakete: Strukturiert. Ein nightly Background-Scan (APScheduler 02:00 UTC) iteriert alle online Kiosk-Clients und führt via Tactical run_command winget export (für die installierte Liste, JSON-Output) und winget upgrade (für verfügbare Updates) als SYSTEM aus. Result landet in der DB-Tabelle agent_winget_state — kein Tactical-Round-Trip mehr nötig wenn der Kiosk-Client /api/v1/packages aufruft. Nach jeder User-Aktion (Install/Update/Uninstall) läuft sofort ein „targeted Re-Scan" für nur diesen einen Agent damit der Status binnen Sekunden frisch ist.
Wichtig: Tactical scannt die installierte Software nicht in Echtzeit, sondern in Intervallen (Minuten bis Stunden). Nach einem Install kann es einen Moment dauern, bis der Status im Kiosk als „Installiert" angezeigt wird. Wer das beschleunigen will, kann in Tactical auf dem Agent manuell "Scan Installed Software" triggern.

Machine-Tokens

Jeder Kiosk-Client bekommt bei der Registrierung ein signiertes JWT (Machine-Token), das in HKLM\SOFTWARE\Softshelf gespeichert wird. Das Token enthält:

  • agent_id — bindet den Token an eine bestimmte Tactical-Agent-ID
  • hostname — für Logging
  • exp — Ablaufzeit (default 365 Tage, via token_ttl_days änderbar; 0 = unbegrenzt)
  • tv — Token-Version, wird bei jedem Re-Register hochgezählt

Das Token wird bei jedem API-Call vom Kiosk-Client mitgeschickt und vom Proxy verifiziert: Signatur + Ablauf + Token-Version gegen die DB. Wenn eines dieser drei fehlt, antwortet der Proxy mit 401:

  • 401 Token abgelaufen — exp ist vorbei. Client muss neu registriert werden.
  • 401 Token widerrufen — tv stimmt nicht mit der DB überein. Passiert wenn der Admin das Gerät zwischenzeitlich erneut deployt hat oder die Token-Version manuell gebumpt wurde.
Der Kiosk-Client selbst kann sich nicht automatisch neu registrieren — er hat kein Registration-Secret. Nach einem Token-Problem musst du den Client erneut ausrollen (via Tactical-Script oder manuell den Installer ausführen).

Wie kommen Dateien zum Client?

Dieser Abschnitt erklärt den vollständigen Weg einer hochgeladenen .msi oder .exe vom Browser des Admins bis zur fertigen Installation auf dem Endgerät — mit allen Stationen, Speicherorten und Sicherheits-Mechanismen. Wichtig zu verstehen wenn du wissen willst wo deine Dateien wirklich liegen und wer sie wann sieht.

Schritt für Schritt

Phase A: Upload (Admin → Server)

  1. Admin wählt eine Datei im Admin-UI und füllt Display-Name, Kategorie aus. Optional auch Install-Args, Detection-Name, Uninstall-Command (bei MSI werden diese auto-generiert).
  2. Browser sendet einen POST /admin/api/upload als multipart/form-data. Die Datei wird in einer Streaming-Verbindung übertragen — der Browser hält sie nicht komplett im Speicher.
  3. Der Proxy-Container empfängt den Stream chunk-weise:
    • Schreibt zunächst nach /app/data/uploads/_tmp_<random>.<ext>
    • Berechnet parallel den SHA-256-Hash über alle Bytes
    • Bricht ab wenn die Datei das Größenlimit (max_upload_mb, default 500 MB) überschreitet
  4. Bei einer MSI: der Proxy ruft msiinfo export <file> Property auf (von msitools, im Image installiert) und extrahiert ProductName, ProductCode, ProductVersion, Manufacturer. Daraus werden Detection-Name und Default-Uninstall-Command vorgeschlagen.
  5. Atomares Rename: nach erfolgreichem Upload wird die temp-Datei nach /app/data/uploads/<sha256>.<ext> umbenannt. Wenn schon eine Datei mit demselben Hash existiert (gleicher Inhalt), wird die temp-Datei verworfen — automatische Deduplikation.
  6. Eintrag in der Whitelist: ein neuer Datensatz wird in die Tabelle packages geschrieben mit:
    • type='custom'
    • sha256 (Hash, dient als Storage-Key)
    • filename (Original-Dateiname für die Anzeige)
    • size_bytes (für Anzeige + Limits)
    • install_args, uninstall_cmd, detection_name

Phase B: Speicherung auf dem Proxy-Host

Der Speicherort /app/data/uploads/ innerhalb des Containers ist ein Bind-Mount auf den Host. Der physische Speicherort ist:

/opt/softshelf/data/uploads/
├── <sha256-1>.msi
├── <sha256-2>.exe
└── ...

Die Dateinamen sind die SHA-256-Hashes des Inhalts plus die ursprüngliche Extension. Eigenschaften dieses Layouts:

  • Deduplikation: Lädst du dieselbe Datei zweimal hoch (z. B. unter zwei verschiedenen Display-Namen), liegt sie nur einmal physisch auf der Platte.
  • Persistent über Container-Restarts: Bind-Mount, daher überleben Dateien docker-compose down + up.
  • Im Backup enthalten: wenn du ./data/ tarst, hast du alle hochgeladenen Files dabei.
  • Eigentümer: Im Container UID 1001 (User softshelf), auf dem Host entsprechend abgebildet. Der Container läuft nicht als root.
  • Kein direkter Internet-Zugriff: die Dateien sind nur über den authentifizierten Download-Endpoint erreichbar — siehe Phase D.

Phase C: Install-Anfrage vom Kiosk-Client

  1. End-User klickt im Kiosk-UI auf "Installieren" für ein custom-Paket.
  2. softshelf.exe sendet POST /api/v1/install mit {"package_name": "..."} und seinem Bearer <JWT>-Header.
  3. Der Proxy verifiziert das Machine-Token (Signatur, Ablauf, token_version), zieht agent_id aus dem Token-Payload.
  4. Lookup in der DB: Paket muss in der Whitelist sein und vom Typ custom. Der gespeicherte SHA-256 wird gelesen.
  5. Proxy generiert einen kurzlebigen Download-Token (eigenständiges JWT, signiert mit SECRET_KEY):
    • typ: "dl"
    • sha: <sha256> — bindet den Token an genau diese Datei
    • agent_id: <agent> — bindet ihn an den anfragenden Agent
    • exp: now + 300 — Token läuft nach 5 Minuten ab
  6. Proxy konstruiert den PowerShell-Command: 
    $tmp = Join-Path $env:TEMP 'kiosk_install_xxxx.msi'
Invoke-WebRequest -Uri 'https://proxy/api/v1/file/<sha>?token=<jwt>' `
                  -OutFile $tmp -UseBasicParsing
Start-Process -FilePath msiexec `
              -ArgumentList '/i', $tmp, '/qn', '/norestart' `
              -Wait -PassThru -NoNewWindow
Remove-Item $tmp -Force -ErrorAction SilentlyContinue
  7. Fire-and-Forget: Der Proxy schickt diesen Command via POST an Tactical's /agents/<id>/cmd/-Endpoint und gibt sofort eine Antwort an den Kiosk-Client zurück ("Installation gestartet"). Der eigentliche Tactical-Call wartet im Hintergrund auf das Resultat — der Kiosk-Client darf nicht 2-5 Minuten am POST hängen, sonst Timeout im UI.

Phase D: Tactical-Agent führt aus

  1. Tactical RMM empfängt den Command via REST und verteilt ihn via NATS an den entsprechenden Agent auf dem Windows-PC.
  2. Der Tactical-Agent (Windows-Service, läuft als NT AUTHORITY\SYSTEM) empfängt die NATS-Nachricht und startet einen PowerShell-Job in einem temporären Working-Directory.
  3. Schritt 1 im Job: Invoke-WebRequest ruft den Proxy-Endpoint /api/v1/file/<sha>?token=<jwt> auf:
    • Der Proxy verifiziert den Token (Signatur, exp, typ, sha-match)
    • Schaut in der DB nach ob das Paket für diesen Hash existiert
    • Schaut auf der Platte ob die Datei unter /app/data/uploads/<sha>.<ext> liegt
    • Streamt die Datei via FastAPI FileResponse mit der ursprünglichen Filename-Header zurück
  4. Der Agent speichert die Datei in %TEMP%\kiosk_install_<random>.<ext> auf dem Windows-PC.
  5. Schritt 2 im Job: Start-Process ruft den Installer mit den definierten Silent-Args auf und wartet bis er fertig ist (-Wait).
    • Bei MSI: msiexec /i $tmp /qn /norestart
    • Bei EXE: $tmp /S (oder was immer in install_args steht)
  6. Exit-Code-Check: 0 oder 3010 = OK, alles andere = Fehler. PowerShell ruft exit entsprechend.
  7. Cleanup: die temporäre Datei wird gelöscht (Remove-Item $tmp -Force).
  8. Result-Push: der Tactical-Agent sendet das Job-Result via NATS zurück an Tactical, der es im Job-History-Log ablegt. Dort kannst du als Admin die Ausgabe sehen wenn was schief geht.

Phase E: Detection — Wann erkennt der Kiosk dass es installiert ist?

  1. Tactical scannt periodisch die installierte Software auf jedem Agent (Intervall meist Minuten bis Stunden, abhängig von Tactical-Konfiguration).
  2. Beim nächsten Refresh im Kiosk-UI fragt der Kiosk-Client GET /api/v1/packages ab.
  3. Der Proxy zieht Tactical's aktuellen Software-Inventar für diesen Agent (GET /software/<agent_id>/).
  4. Substring-Match: für jedes custom-Paket wird der gespeicherte detection_name bidirektional gegen die Windows-Anzeigenamen aus Tactical's Inventar verglichen.
  5. Bei Treffer: Paket wird im Kiosk-UI als „Installiert" angezeigt mit der von Tactical gemeldeten Version.

Sicherheits-Eigenschaften der Download-URL

Die Download-URL für custom-Dateien sieht so aus:

https://proxy.intern:8765/api/v1/file/821415899fda40063b...?token=eyJhbGciOiJIUzI1NiI...

Sie ist kein öffentlicher Link. Folgende Schutzmechanismen greifen:

  • HMAC-Signatur: der Token ist mit dem Proxy-SECRET_KEY signiert. Niemand kann einen gültigen Token fälschen ohne den Schlüssel zu kennen.
  • Hash-Bindung: der Token enthält den SHA-256 der Datei. Der Proxy prüft beim Download dass URL-Pfad und Token-Hash übereinstimmen — der Token kann nicht für andere Dateien wiederverwendet werden.
  • Agent-Bindung: der Token enthält die agent_id des anfragenden PCs. Wenn ein Angreifer die URL aus dem Tactical-Audit-Log zieht, ist sie immer noch nur für diesen einen Agent-Kontext semantisch sinnvoll (keine Generalisierung möglich).
  • Kurzes Ablauf-Fenster: 5 Minuten zwischen Generierung und Ablauf. Lang genug für den Download (auch bei großen MSIs auf langsamen Verbindungen), kurz genug um Replay-Attacks praktisch unmöglich zu machen.
  • Nicht-bruteforce-bar: der Token-Pfad enthält 256 Bit Entropie aus dem JWT — niemand rät den richtigen Token in 5 Minuten.
  • Audit-Log: jeder Aufruf wird im Audit-Log protokolliert (Method, Path, Client-IP, Status, Dauer). Du siehst im Tab Audit-Log alle Download-Requests.
Aktuell ist HTTPS noch nicht aktiviert — die URL fliegt im Klartext durch das interne Netz. Ein Angreifer im selben LAN könnte sie binnen der 5 Minuten Ablauffrist mitschneiden und die Datei einmal abrufen (Inhalt ist eh schon im LAN auf dem Ziel-PC). Sobald TLS via Reverse-Proxy davor gesetzt ist, ist auch dieser Vektor zu.

Was passiert beim Löschen eines custom-Pakets?

  1. Du klickst auf das × auf der Karte und bestätigst den Confirm-Dialog.
  2. Der Proxy ruft DELETE /admin/api/enable/<name>.
  3. Der Proxy lädt das Paket aus der DB, merkt sich den sha256-Hash.
  4. Der DB-Eintrag wird gelöscht.
  5. Safety-Check: der Proxy fragt SELECT COUNT(*) FROM packages WHERE sha256 = ? — gibt es noch andere Pakete die dieselbe Datei referenzieren?
  6. Wenn COUNT = 0: die Datei wird physisch von /app/data/uploads/ gelöscht.
  7. Wenn COUNT > 0: die Datei bleibt liegen — ein anderes Paket nutzt sie noch.
Bereits installierte Software auf Client-PCs wird vom Delete-Vorgang nicht berührt. Der Delete entfernt nur den Eintrag aus der Self-Service-Whitelist und ggf. die Quelldatei vom Proxy-Speicher. Wenn du die Software auch von den PCs runter haben willst, müssen die End-User vorher im Kiosk auf "Deinstallieren" klicken (oder du machst es via Tactical direkt).

Paket-Workflows

Chocolatey-Paket freigeben

  1. Tab Pakete öffnen
  2. Oben rechts + Paket hinzufügen klicken
  3. Im Slide-in-Panel sicherstellen dass der Reiter Aus Chocolatey aktiv ist
  4. Im Suchfeld den Paketnamen eintippen (z. B. firefox, vscode, 7zip). Die Suche ist debounced — Treffer erscheinen nach ca. 300 ms. Gesucht wird in Tactical's lokal gecachter Chocolatey-Liste.
  5. In der Trefferliste auf das +-Icon klicken, um das Paket freizugeben. Es erscheint sofort in der Whitelist mit Default-Kategorie Allgemein.
  6. Bei Bedarf:
    • Anzeigename über das Input-Feld auf der Karte anpassen
    • Kategorie über die Kategorie-Pille ändern
Der +-Button wird grau & ein grüner Haken erscheint wenn das Paket bereits freigegeben ist. So vermeidest du Dubletten.

Winget-Paket freigeben

Für Software aus dem offiziellen Microsoft-winget-Catalog. Im Vergleich zu Chocolatey ein deutlich grösserer Katalog (Browser, Entwickler-Tools, Microsoft-Apps, Vendor-Tools wie Zoom oder Dropbox), und die Detection läuft strukturiert über die winget-PackageIdentifier statt per Substring- Heuristik. Updates erkennt das System automatisch via winget upgrade.

  1. Tab Pakete → + Paket hinzufügen
  2. Im Slide-in-Panel auf den Reiter Aus Winget wechseln
  3. Im Suchfeld den Paketnamen oder den Hersteller eintippen (z. B. firefox, vscode, vlc, mozilla). Die Suche läuft gegen einen lokalen SQLite-Mirror der Microsoft-Catalog-Datei (cdn.winget.microsoft.com/cache/source.msix) — kein Tactical-Round-Trip, kein Internet-Call beim Suchen, Treffer in unter 100 ms. Der Mirror wird täglich um 01:30 UTC frisch gezogen.
  4. Trefferliste zeigt pro Eintrag: Anzeige-Name, Publisher, neueste Version (semver-aware), winget-PackageIdentifier. Sortiert nach Match-Qualität (Last-Segment-Match > Substring-Match).
  5. Auf das +-Icon klicken um das Paket freizugeben. Es wandert sofort in die Whitelist als type='winget' mit der PackageIdentifier als name.
  6. Bei Bedarf: Anzeigename und Kategorie wie bei choco anpassen.
winget-Pakete brauchen winget auf dem Endgerät. Auf Windows 11 ist der „App Installer" mit winget standardmäßig installiert. Auf älteren Windows-10- Builds und auf LTSC fehlt er und muss separat ausgerollt werden. Der nightly Scan erkennt fehlendes winget und markiert den Agent in agent_scan_meta mit last_status='no_winget' — sichtbar in der Agent-Detail-Page als roter Hinweis.
Per-User-Apps und Microsoft-Store-Apps: winget unter SYSTEM-Kontext kann Pakete die per-user installiert wurden meist nicht entfernen oder upgraden. Symptom beim Uninstall: ExitCode -1978335162 (NO_UNINSTALL_INFO_FOUND). Beim Upgrade: No available upgrade found obwohl winget upgrade ohne --id die Update-Verfügbarkeit anzeigt. Workaround: das Paket manuell als logged-in User upgraden, oder einen anderen Installer nutzen.

Eigene MSI hochladen

Für Software die nicht im Chocolatey-Repo ist (z. B. interne Tools, proprietäre Vendor-Installer, Softshelf-Eigenentwicklungen).

  1. Tab Pakete → + Paket hinzufügen
  2. Im Panel auf den Reiter Eigenes Paket wechseln
  3. Datei auswählen oder in die gestrichelte Dropzone ziehen. Maximale Größe: 500 MB (änderbar in den Einstellungen).
  4. Der Anzeigename wird automatisch aus dem Dateinamen abgeleitet — ggf. anpassen
  5. Kategorie eintragen (frei wählbar, z. B. "Softshelf-Tools")
  6. Die nachfolgenden Felder können bei MSIs leer bleiben — der Proxy extrahiert beim Upload automatisch:
    • Detection-Name aus ProductName der MSI
    • Uninstall-Command aus ProductCode der MSI: msiexec /x "{GUID}" /qn /norestart
    • Install-Args default auf /qn /norestart
  7. Hochladen klicken. Ein Progress-Bar zeigt den Upload-Fortschritt.
  8. Nach erfolgreichem Upload erscheint das Paket in der Whitelist mit dem Tag Eigenes Paket.
Die MSI-Metadaten-Extraktion läuft server-seitig via msiinfo (aus dem msitools-Paket). Funktioniert mit allen Standard-MSI-Installern von Microsoft, NullSoft, InstallShield, Advanced Installer etc.

Eigene EXE hochladen

Gleicher Ablauf wie bei MSI, mit einem Unterschied: die Metadaten-Extraktion funktioniert nicht. Silent-Install-Flags und Detection-Name musst du manuell eintragen oder nachträglich ermitteln.

  1. Datei hochladen (Ablauf wie oben)
  2. Install-Args auf den Silent-Flag des Installers setzen:
    • NSIS-Installer: /S (großes S)
    • InnoSetup: /VERYSILENT /SUPPRESSMSGBOXES /NORESTART
    • Squirrel: --silent
    • Selbst-gewrapped MSI-Setup: oft /quiet oder -q
    • Unbekannt? Im Zweifel die Herstellerdoku oder --help auf der Kommandozeile versuchen
  3. Detection-Name: das ist exakt der Name wie das Programm nach Installation in Windows „Apps & Features" auftaucht. Tipp: Paket erst einmal testhalber auf einer Test-Maschine installieren, in „Apps & Features" den Namen kopieren, dann im Edit-Panel eintragen.
  4. Uninstall-Command: kann manuell eingetragen werden oder nachträglich aus Tactical gezogen werden (siehe Uninstall aus Tactical übernehmen).
Wenn der Silent-Flag nicht stimmt, öffnet der Installer einen GUI-Dialog auf dem Ziel-PC, der dort niemand sieht — die Installation hängt, und der Agent bekommt einen Timeout. Teste Silent-Flags einmal manuell auf deiner Test-Maschine bevor du sie einem custom-Paket hinterlegst.

Programm-Ordner hochladen

Für Software, die mehr braucht als eine einzelne MSI/EXE — z.B. InstallShield mit externen .cab-Dateien, Programme mit Konfig-Files daneben, oder Custom- Installer mit DLLs/Skripten im selben Verzeichnis. Der ganze Ordner wird hochgeladen, serverseitig zu einem ZIP gepackt und zentral abgelegt — beim Install lädt der Client das ZIP, entpackt es und startet eine Entry-Point-Datei im Ordner.

  1. Tab Pakete → + Paket hinzufügen → Eigenes Paket
  2. Im Sub-Toggle auf Programm-Ordner wechseln
  3. Ordner wählen klicken → Browser-Dialog: kompletten Ordner auswählen (alle Unterverzeichnisse werden mit aufgenommen)
  4. Der Entry-Point-Dropdown wird automatisch befüllt: alle .exe, .msi, .bat und .cmd aus dem Ordner sind wählbar. Vorausgewählt wird die wahrscheinlichste Installer-Datei (Priorität: setup.exe → installer.exe → install.* → flachster .exe-Pfad).
  5. Anzeigename eingeben (wird im Kiosk gezeigt). Wird aus dem Ordnernamen vorgefüllt.
  6. Argumente für den Installer bei Bedarf — die werden 1:1 an die Entry-Point-EXE weitergereicht (z.B. /silent oder /qn /norestart).
  7. Detection-Name (optional) — bei Programm-Ordnern muss das manuell gesetzt werden, da kein MSI-ProductName auto-extrahiert werden kann.
  8. Uninstall-Command (optional) — z.B. "C:\Programme\Toolkit\unins000.exe" /VERYSILENT
  9. Hochladen klicken
Was auf dem Client passiert:
  1. Tactical-Agent lädt das ZIP über die signierte Download-URL (5 min TTL, agent-gebunden)
  2. PowerShell entpackt es nach $env:TEMP\kiosk_install_<random>\
  3. Die Entry-Point-EXE wird mit dem Ordner als Arbeitsverzeichnis gestartet (WorkingDirectory) — damit findet der Installer relative Dateien
  4. Nach Abschluss werden ZIP und entpacktes Verzeichnis komplett gelöscht
Größenlimit: Default 500 MB pro Upload. Anpassbar in Einstellungen → max_upload_mb. Der freie Speicherplatz auf der Server-Platte wird im Upload-Panel und im Verteilung-Tab angezeigt.

Den Entry-Point kannst du nachträglich im Edit-Panel ändern (Dropdown mit allen Dateien aus der current-Version). Eine neue Version eines Programm-Ordner-Pakets lädst du ebenfalls im Edit-Panel hoch — der Versions-Upload-Widget passt sich automatisch an Programm-Ordner-Pakete an (Ordnerwahl statt Datei).

Paket nachträglich bearbeiten

Choco-Pakete: Anzeigename direkt auf der Karte editieren (in das Feld klicken, neuen Text eingeben, Feld verlassen → Auto-Save). Kategorie via Klick auf die Kategorie-Pille unter dem Namen.

Custom-Pakete bekommen einen Stift-Button neben dem ×. Klick darauf öffnet das Edit-Panel mit allen Feldern:

  • Anzeigename
  • Kategorie
  • Silent-Install Argumente
  • Detection-Name (inkl. "Aus Tactical holen"-Option siehe unten)
  • Uninstall-Command
  • Versionen — Liste aller Versionen, neue hochladen, andere als aktiv setzen (siehe nächster Abschnitt)
  • Installiert auf — Liste aller Geräte die das Paket über das Self-Service-Center installiert haben, mit der jeweils installierten Version

Der Dateiname, die Dateigröße und der Hash der aktuellen Version sind nicht editierbar — die sind an die physisch hochgeladene Datei gebunden. Für eine neue Version eines custom-Pakets siehe Abschnitt Versionierung & Updates pushen.

Versionierung & Updates pushen

Jedes custom-Paket kann mehrere Versionen haben. Genau eine ist die aktive Version — neue Installs greifen immer darauf zu. Alte Versionen bleiben im Storage liegen, damit du auf eine ältere Version zurückrollen kannst.

Zusätzlich trackt der Proxy welcher Client welche Version installiert hat. Wenn eine neue Version aktiv gesetzt wird, sehen alle Clients die noch auf der älteren Version hängen als „veraltet" markiert — und du kannst mit einem Klick ein Update auf alle veralteten Geräte pushen.

Wie das Tracking funktioniert

  • Beim erfolgreichen Install (sowohl normaler Install als auch gepushtes Update) trägt der Proxy in die Tabelle agent_installations ein: (agent_id, package_name, version_id).
  • Beim Uninstall wird der Eintrag entfernt.
  • Tracking läuft nur für custom-Pakete, nicht für Chocolatey (Chocolatey hat seinen eigenen Update-Mechanismus).
  • Wenn ein Install fehlschlägt (z. B. Tactical-Cmd timeout), wird nicht getrackt — der Eintrag landet erst bei tatsächlichem Erfolg.

Neue Version hochladen

  1. Edit-Panel des custom-Pakets öffnen (Stift-Icon auf der Karte).
  2. Im Abschnitt Versionen auf + Neue Version hochladen klicken.
  3. Datei wählen (.msi oder .exe). Der Versions-Label wird automatisch als v2, v3 etc. vorgeschlagen — du kannst aber auch z. B. 1.2.3 oder 2026-Q2 setzen, solange das Label innerhalb des Pakets eindeutig ist.
  4. Optional: Notizen eingeben (z. B. Changelog, Bugfix-Beschreibung).
  5. Checkbox Sofort aktiv setzen:
    • Aktiviert (Default): die neue Version wird sofort die aktive Version. Alle zukünftigen Installs nutzen sie. Bestehende Installationen werden als „veraltet" markiert.
    • Deaktiviert: die Version wird nur hochgeladen und gespeichert. Aktiv bleibt die alte. Praktisch zum Vorbereiten / Testen einer neuen Version, bevor man sie ausrollt.
  6. Hochladen klicken. Datei wird hochgeladen, gehasht, in der Versionsliste angezeigt.
Die Install-Argumente und das Uninstall-Command werden für die neue Version vom bestehenden Paket übernommen — du musst sie nicht erneut eingeben. Falls du sie für die neue Version anders haben willst, erst hochladen, dann im Edit-Panel anpassen und speichern.

Versions-Übersicht

Die Versions-Liste zeigt für jede Version:

  • Label als Badge (die aktive Version ist dunkel hervorgehoben)
  • Dateiname und Größe
  • Upload-Datum und Notizen
  • Buttons:
    • Aktiv setzen — macht diese Version zur neuen current. Synchronisiert die flachen Felder im packages-Eintrag (filename, sha256, install_args, uninstall_cmd) mit dieser Version.
    • Löschen — entfernt die Version permanent. Nicht möglich für die aktive Version — vorher eine andere als aktiv setzen. Beim Löschen wird die physische Datei nur dann gelöscht, wenn keine andere Version oder kein anderes Paket denselben SHA-256-Hash referenziert (Shared-File-Schutz).

Update auf veraltete Geräte pushen

  1. Im Edit-Panel zeigt die Versions-Übersicht oberhalb der Liste eine Statistik: X installiert · Y aktuell · Z veraltet. Auf den Paket-Karten siehst du zusätzlich ein orangefarbenes Badge wenn das Paket veraltete Installationen hat.
  2. Im Abschnitt Installiert auf siehst du jede einzelne Maschine mit ihrer installierten Version. Veraltete Versionen sind gelb markiert.
  3. Wenn es mindestens 1 veraltetes Gerät gibt, wird der Button Update pushen unten im Panel aktiv. Klick darauf zeigt die Anzahl betroffener Geräte.
  4. Bestätigen mit Pushen. Der Proxy:
    • baut für jeden veralteten Agent das Install-Command der aktuellen Version,
    • schickt das via Tactical-Run-Command an den Agent,
    • läuft fire-and-forget im Hintergrund (du brauchst nicht zu warten).
  5. Sobald die Installation auf einem Agent erfolgreich durchläuft, wird sein agent_installations-Eintrag auf die neue Version aktualisiert. Veraltet- Status verschwindet.
Push ist ein Reinstall, kein Update-im-engeren-Sinn. Was tatsächlich passiert, hängt vom Installer ab:
  • MSI mit gleichem ProductCode → MSI führt ein „Major Upgrade" oder „Minor Upgrade" durch (je nach ProductVersion). Funktioniert sauber.
  • MSI mit anderem ProductCode → MSI versucht side-by-side zu installieren. Du brauchst vorher einen Uninstall der alten Version oder das neue MSI muss über die UpgradeCode- Logik korrekt vorgehen.
  • EXE/NSIS/InnoSetup → der Installer entscheidet selbst. Die meisten erkennen die alte Version und überschreiben sie.
Im Zweifel: vorher einen Test-Agent updaten und prüfen, bevor du den großen Push fährst.
Was passiert mit Geräten ohne Tracking-Eintrag?
Geräte die das Paket vor der Tracking-Einführung installiert haben oder bei denen der Install nie über das Self-Service-Center lief, haben keinen agent_installations- Eintrag. Sie tauchen weder als „aktuell" noch als „veraltet" auf — das Paket weiß nichts von ihnen. Push-Update ignoriert sie. Sobald sie das Paket einmal über den Kiosk (de)installieren, sind sie im Tracking.

Verteilung-Tab

Der Tab Verteilung ist die zentrale Übersicht aller custom-Pakete und ihrer Verteilung über alle Kiosk-Clients. Im Gegensatz zur Edit-Panel-Versions-Sektion (paket-zentriert) ist das hier eine flache Liste, optimiert für den Workflow „auf welchem Gerät ist welche Version, und was muss ich aktualisieren oder entfernen".

Was du dort siehst

Pro custom-Paket eine Karte mit Header und Tabelle:

  • Header: Anzeigename, Datei, Größe, current-Version-Label, Statistik (X Geräte · Y aktuell · Z veraltet) und Buttons.
  • Push-Button: erscheint nur wenn es veraltete Geräte gibt — pusht die current Version auf alle veralteten Geräte (gleicher Effekt wie der Push-Button im Edit-Panel).
  • Bearbeiten-Button: wechselt zum Pakete-Tab und öffnet das Edit-Panel des Pakets.
  • Tabelle pro Gerät: Hostname, installierte Version (gelb wenn veraltet, grün wenn aktuell), Install-Datum und zwei Aktionen:
    • Updaten — nur sichtbar wenn das Gerät auf einer veralteten Version ist. Triggert einen einzelnen Reinstall der current-Version auf nur diesem Gerät.
    • Entfernen — triggert ein Uninstall über das im Paket hinterlegte Uninstall-Command. Disabled wenn kein Uninstall-Command gesetzt ist.

Filter

Die Suchleiste oben filtert sowohl nach Paketname als auch nach Hostname/Agent-ID. Beispiel: Tippst du pc-01 ein, werden alle Karten angezeigt in denen mindestens ein Gerät passt — und in der Tabelle siehst du nur die passenden Geräte. So findest du schnell „was hat dieser Rechner alles über das Self-Service-Center bekommen".

Tracking-Hinweis

Im Verteilung-Tab tauchen nur Geräte auf, die das Paket über das Self-Service-Center installiert haben (Tracking via agent_installations). Geräte, auf denen die Software auf anderem Weg installiert wurde, sind unbekannt — sie werden weder als „aktuell" noch als „veraltet" angezeigt und ignorieren den Push-Update-Button. Die Erfassung beginnt erst nachdem ein Install/Uninstall einmal über den Kiosk gelaufen ist.

Fleet-Discovery (Winget)

Das Discovery-Feature erkennt automatisch welche winget-Pakete auf deinen Endgeräten installiert sind aber noch nicht in der Softshelf-Whitelist stehen, und bietet ein-Klick-Aktivierung an. So kannst du eine bestehende Flotte sukzessive in Softshelf reinholen ohne jedes Paket einzeln im Catalog zu suchen.

Wo es auftaucht

  1. Im Header rechts neben Version und User-Menu erscheint ein Discovery-Banner sobald die Flotte mindestens ein unverwaltetes winget-Paket hat. Beispiel: „12 Pakete in der Flotte unverwaltet".
  2. Klick auf den Banner öffnet das Aktivieren-Panel direkt mit dem Winget-Tab und expandiertem Discovery-Block.
  3. Alternativ: Tab Pakete → + Paket hinzufügen → Reiter Aus Winget → unten den Block „In der Flotte gefunden" ausklappen.

Was angezeigt wird

Pro Discovery-Eintrag siehst du:

  • Display-Name (z. B. „Mozilla Firefox") — wie es in „Apps & Features" auf den Endgeräten heisst
  • Source-Tag: winget wenn der Eintrag aus dem nightly winget-Scan kommt (kanonisch, sicher matchbar), oder software-scan wenn er aus dem Tactical-Software-Scan stammt und per Confidence-Heuristik gegen den winget-Catalog gemappt wurde
  • Confidence-Badge: sicher / wahrscheinlich / unsicher / kein Match. Bei Source winget ist die Confidence immer sicher, weil die ID direkt aus winget kommt
  • Install-Count: auf wie vielen Geräten der Flotte das Paket installiert ist
  • winget-PackageIdentifier: die ID die beim Aktivieren in die Whitelist landet
  • Aktion: +-Button bei Confidence ≥ unsicher, oder „manuell"-Hinweis bei kein Match

Wie die zwei Quellen kombiniert werden

Discovery hat zwei parallele Quellen die zusammen gemerged werden:

  1. Primary: agent_winget_state. Aus dem nightly winget-Scan (siehe Abschnitt Grundkonzepte) sind die installierten Pakete pro Agent bereits mit kanonischer winget-PackageIdentifier in der DB. Eine simple SQL-Query findet alle winget_id-Werte die in keinem packages-Eintrag auftauchen — das sind die Discovery-Kandidaten mit Confidence sicher.
  2. Bonus: discovery_enrichment. Ein zweiter Background-Job (Enrichment, läuft täglich um 02:30 UTC) holt von Tactical pro online Agent die installierte Software-Liste (das ist die ARP-basierte Sicht, also auch Software die NICHT via winget installiert wurde) und versucht jeden distinct Display-Namen gegen den lokalen winget-Catalog zu matchen. Token-basierte Score-Heuristik mit Generic- Filter (Wörter wie 64bit, x64, pro, enterprise, de werden ignoriert um false-positive Matches zu vermeiden). Ergebnis: discovery_enrichment-Tabelle mit Display-Name, gefundener winget-ID, Confidence und Install-Count.
Der Enrichment-Job läuft asynchron einmal täglich. Wenn du gerade einen neuen Agent ausgerollt hast, wird seine installierte Software erst beim nächsten Enrichment-Run in der Discovery sichtbar. Manueller Trigger via POST /admin/api/winget/run-enrichment.

Uninstall aus Tactical übernehmen

Das macht es einfach, den korrekten Uninstall-Command eines custom-EXE-Installers zu finden: statt mühsam die Registry durchwühlen, hat Tactical den Wert bereits in seinem Software-Inventar.

  1. Paket testweise auf mindestens einem Kiosk-Client installieren (via Install-Klick im Kiosk-UI).
  2. Warten bis Tactical's nächster Software-Scan durchgelaufen ist (einige Minuten bis Stunden).
  3. Im Admin-UI: Edit-Panel des custom-Pakets öffnen (Stift-Button auf der Karte).
  4. Neben dem Feld Uninstall-Command auf Aus Tactical holen klicken.
  5. Der Proxy scannt die letzten 30 aktiven Agents, sucht nach dem detection_name, und übernimmt bei Treffer die UninstallString aus der Windows-Registry.
  6. Der gefundene Wert wird ins Feld geschrieben, der Toast zeigt welcher Agent getroffen wurde.
  7. Meist musst du noch Silent-Flags ergänzen (der Registry-Wert ist oft nicht silent):
    • MSI-Apps: /X{GUID} → /X{GUID} /qn /norestart
    • NSIS-Apps: "...\uninstall.exe" → "...\uninstall.exe" /S
    • InnoSetup: "...\unins000.exe" → "...\unins000.exe" /VERYSILENT /NORESTART
  8. Speichern klicken.
Der Scan läuft sequentiell durch die 30 zuletzt aktiven Agents und bricht beim ersten Treffer ab. Dauer: wenige Sekunden bis max ~30 Sekunden, abhängig von der Tactical-API-Latenz und der Position deines Test-Agents in der Liste.

Paket löschen

  1. Auf der Karte das ×-Icon klicken
  2. Im Confirm-Modal bestätigen

Was dabei passiert:

  • Choco-Paket: Entfernung aus der Whitelist. Bereits auf Clients installierte Software bleibt dort installiert — der Delete-Button entfernt nur den Eintrag bei uns, nicht die Software auf den Maschinen.
  • Custom-Paket: Whitelist-Eintrag plus alle Versionen, das komplette Installations-Tracking, und für jede Version die physische Datei aus dem Upload-Storage — sofern kein anderes Paket oder keine andere Version denselben SHA-256-Hash referenziert (Shared-File-Schutz). Bereits auf Clients installierte Software bleibt ebenfalls dort.
Der Delete entfernt nicht die Software vom Client-PC! Dafür müsste der End-User im Kiosk-UI vor dem Löschen auf "Deinstallieren" klicken, oder ein Admin muss das via Tactical direkt ausführen.
Neue Version eines bestehenden Pakets: nicht löschen + neu anlegen — sondern im Edit-Panel im Abschnitt Versionen die neue Datei hochladen. Siehe Versionierung & Updates pushen.

Agent-Detail-Page: Software pro Client

Im Tab Kiosk-Clients öffnet ein Klick auf eine Zeile die Agent- Detail-Page als Vollbild-Sicht (ersetzt das frühere Slide-in-Panel). Sie ist die zentrale Stelle für „was ist eigentlich auf diesem PC und was kann ich damit machen".

Header-Toolbar

Oben über dem Detail-Bereich:

  • ← Zurück — schließt die Detail-Sicht und zeigt wieder die Agent-Liste.
  • Hostname + Agent-ID — bei gesperrten Agents mit Suffix · GESPERRT.
  • Neu scannen — triggert sofort einen winget-Re-Scan auf diesem Agent. Setzt zusätzlich consecutive_failures in agent_scan_meta auf 0, falls der Agent durch frühere Fehler temporär aus dem nightly Batch ausgeschlossen war.
  • Token widerrufen / Sperren / Löschen — siehe Agent-Lifecycle für die Wirkung jeder Aktion.

Installierte Software

Die Hauptliste zeigt die vereinte Sicht aus zwei Datenquellen:

  1. Tactical software-scan (ARP-basiert, alle installierten Programme aus der Windows-Registry)
  2. agent_winget_state (kanonische winget-Inventur aus dem nightly Scan: PackageIdentifier + installed_version + available_version)

Die beiden Quellen werden serverseitig zusammen gemerged und dedupt. Wenn Tactical „Mozilla Firefox" sieht UND der winget-Scan Mozilla.Firefox sieht, taucht das in der Liste als EINE Zeile mit beiden Datensätzen auf. Das Matching läuft über eine Token-Score-Heuristik gegen die winget-PackageIdentifier- Segmente, mit Filter für generische Tokens (64bit, x64, pro, …) damit z. B. „Adobe Acrobat (64-bit)" nicht versehentlich „SAP GUI 64bit" matcht.

Pro Software-Zeile siehst du:

  • Display-Name (groß)
  • Status-Badges: verwaltet · winget / verwaltet · choco / verwaltet · eigen wenn das Paket in der Softshelf-Whitelist ist; winget · unverwaltet für winget-Pakete ohne Whitelist-Eintrag; unverwaltet für tactical-only Software
  • Update-Badge: Update X.Y.Z wenn der nightly winget-Scan eine neuere Version gefunden hat
  • Sub-Zeile: winget-PackageIdentifier (falls vorhanden), installierte Version, Publisher

Aktionen pro Zeile

StatusVerfügbare ButtonsWas passiert
verwaltet + Update verfügbar Updaten + Deinstallieren Updaten startet winget upgrade --id … als SYSTEM. Deinstallieren startet den passenden Uninstall-Pfad je nach Typ.
verwaltet + aktuell Deinstallieren Type-Dispatch: Choco-Uninstall, Custom-Uninstall, oder winget uninstall --force

Wie der Status nach einer Aktion frisch wird

Sobald du auf Updaten, Deinstallieren, Aktivieren oder Entfernen klickst, wird die Zeile mit einem Spinner gesperrt. Das UI pollt im Hintergrund alle 5 Sekunden den Status-Endpoint und vergleicht scan_meta.last_scan_at gegen einen Snapshot vom Klick-Zeitpunkt. Sobald der targeted Re-Scan auf dem Agent durch ist und der Timestamp sich bewegt, refresht die Sicht automatisch — du musst nicht F5 drücken. Hartstop bei 6 Minuten.
Wichtig zur winget-Performance: Eine Update- oder Install-Aktion via winget kann je nach Paket-Größe zwischen wenigen Sekunden und mehreren Minuten dauern (Download + Installer-Run). Während dieser Zeit zeigt der Spinner „läuft …" und die Zeile bleibt gesperrt. Container-Restart in genau diesem Zeitfenster killt den Background-Task und verhindert den nachgelagerten Re-Scan — der Agent läuft zwar fertig durch, aber die DB-Sicht bleibt stale bis du Neu scannen drückst.

Installations-Verlauf

Unter der Software-Liste folgt der bisherige Installations-Verlauf aus install_log für diesen Agent: jede Install/Uninstall-Aktion (von End-User über Kiosk-Client oder von Admin über Detail-Page) ist da als Audit- Eintrag mit Zeitstempel sichtbar.

Agent-Lifecycle: Revoke / Löschen / Sperren

In der Agent-Detail-Page sind oben in der Header- Toolbar drei Aktionen, die unterschiedlich harte Konsequenzen haben. Wichtig: alle drei sind kombinierbar — die typische Reihenfolge bei einem kompromittierten Gerät ist Sperren → Löschen.

AktionWas passiertRe-Register?
Token widerrufen Bumpt token_version in der DB. Der laufende Machine-Token wird beim nächsten Request mit 401 abgelehnt. Der Agent bleibt sonst unverändert in der DB (Tracking, Install-Log). Ja — Setup-Script mit gültigem Registration-Secret macht den Client wieder funktionstüchtig.
Löschen Entfernt den Agent komplett aus dem Self-Service Center: agent_installations, install_log und agents-Row. Tokens werden dadurch automatisch ungültig. Die installierte Software auf dem Gerät bleibt erhalten. Ja — solange das Gerät noch existiert und das Setup-Script ausgeführt wird, taucht der Agent wieder auf.
Sperren Setzt den Agent auf die agent_blocklist. Bumpt zusätzlich token_version für sofortige Wirkung. Der Eintrag überlebt ein Löschen des Agents. Nein — /api/v1/register weist den agent_id mit 403 ab, auch wenn das Setup-Script korrekt läuft. Bis zum Entsperren kommt der Client nicht mehr ans Self-Service Center.

Wann was?

  • Token widerrufen: wenn du nur den Token rotieren willst (z.B. weil du nach einem Audit den Verdacht hast, dass die Registry leakt). Der PC bleibt grundsätzlich vertrauenswürdig — er muss nur einmal neu registriert werden.
  • Löschen: wenn der PC physisch nicht mehr existiert oder neu aufgesetzt wird. Saubert die DB auf, der Eintrag taucht in der Liste nicht mehr auf.
  • Sperren: wenn der PC kompromittiert ist oder aus anderen Gründen NIE mehr ans Self-Service Center darf — auch nicht nach einem manuellen Re-Deployment. Klassisches Szenario: gestohlenes Notebook, Mitarbeiter verlässt die Firma und das Gerät bleibt im Feld, etc.

Sperre aufheben (Unban)

Im Detail-Panel eines bereits gesperrten Agents steht statt Sperren ein Entsperren-Button. Klick → der Eintrag aus agent_blocklist wird entfernt und der Client darf sich beim nächsten Re-Register wieder anmelden. Token-Version wird dabei NICHT zurückgesetzt — der Client braucht trotzdem ein frisches Setup-Script-Run.

Sperren überlebt Re-Register: die Blocklist arbeitet auf der agent_id (vergeben von Tactical RMM für jede physische Maschine). Wenn Tactical das Gerät neu registriert, bekommt es eine neue agent_id — und die alte Sperre greift nicht mehr für das frische Tactical-Asset. Wer also wirklich verhindern will dass ein Gerät zurückkommt, muss das auch in Tactical RMM vom Inventar nehmen.
Audit-Trail: Die Blocklist speichert banned_by (Username des Admins) und banned_at (Zeitstempel). Damit ist nachvollziehbar wer wann wen gesperrt hat. Plus optional eine freie reason- Begründung die der Admin beim Sperren eintippen kann.

Kiosk-Client auf PCs ausrollen

Damit ein Windows-PC das Self-Service Center nutzen kann, muss softshelf.exe installiert und beim Proxy registriert sein. Das kann auf drei Arten passieren:

Variante A: Automatisch via Tactical RMM (empfohlen)

  1. Sicherstellen dass auf dem Proxy unter Einstellungen Proxy Public URL gesetzt ist.
  2. EXEs bauen klicken (falls die EXEs noch nicht existieren oder die Proxy-URL sich geändert hat).
  3. In Tactical Settings → Script Manager → New Script anlegen:
    • Name: Deploy Self-Service Center
    • Shell: Powershell
    • Category: Deployment (o.ä.)
    • Script Body: den Code aus dem nächsten Block einfügen
    • Script Arguments: kopier die vorausgefuellte Zeile aus dem Script Arguments-Block weiter unten in das Tactical-Feld unter dem Script-Body. Die Werte stammen direkt aus deinen aktuellen Admin-Einstellungen — wenn du das Registration-Secret rotierst, einfach den Hilfe-Tab neu laden.
  4. Script auf die gewünschten Agents ausführen (Agents-Übersicht → Auswahl → Run Script). Der Rest läuft ohne User-Interaction:
    • Tactical lädt die softshelf-setup.exe vom Proxy
    • Installer läuft silent als SYSTEM
    • Registriert beim Proxy, bekommt Machine-Token
    • Schreibt Registry-Keys nach HKLM
    • Setzt Autostart für alle User
    • Startet softshelf.exe direkt (wenn ein User eingeloggt ist)

Diese Zeile in das Script Arguments-Feld unter dem Script-Body eintragen. Die Werte sind aus den aktuellen Admin-Einstellungen vorausgefuellt — Hilfe neu laden falls du gerade etwas geaendert hast:

PowerShell-Script (copy-paste in Script Body)

Script 1:1 in den Tactical Script-Body packen. Es ist identisch zu installer/deploy_via_trmm.ps1 im Proxy-Repo. --agent-id wird nicht uebergeben — der Installer liest die Tactical-Agent-ID selbst aus HKLM\SOFTWARE\TacticalRMM\agentid.

<#
.SYNOPSIS
    Softshelf - Deployment Script
    Laedt den Installer herunter und fuehrt ihn still aus.
#>
param(
    [Parameter(Mandatory=$true)] [string]$ProxyUrl,
    [Parameter(Mandatory=$true)] [string]$RegistrationSecret,
    [Parameter(Mandatory=$true)] [string]$SetupExeUrl
)

$ErrorActionPreference = "Stop"
$ProgressPreference    = "SilentlyContinue"

# TLS 1.2 erzwingen — sonst schlaegt Invoke-WebRequest auf aelteren Windows-
# Builds bei modernen Reverse-Proxies fehl.
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12

# Temp-Pfade. Der Error-Log-Name muss zum slug passen den setup.py schreibt
# (sonst sieht man bei einem Fehler nur "Unbekannter Fehler").
$SetupFile = [System.IO.Path]::GetFileName(([Uri]$SetupExeUrl).AbsolutePath)
if ([string]::IsNullOrWhiteSpace($SetupFile)) { $SetupFile = "setup.exe" }
$TempSetup = Join-Path $env:TEMP $SetupFile
$Slug      = [System.IO.Path]::GetFileNameWithoutExtension($SetupFile) -replace '-setup$',''
$ErrorLog  = Join-Path $env:TEMP ("{0}_setup_error.txt" -f $Slug.ToLower())

Write-Host "=== Deployment ===" -ForegroundColor Cyan
Write-Host "Proxy: $ProxyUrl"
Write-Host "Setup: $SetupFile"
Write-Host ""

# Installer herunterladen
Write-Host "Lade $SetupFile herunter..."
try {
    Invoke-WebRequest -Uri $SetupExeUrl -OutFile $TempSetup -UseBasicParsing
} catch {
    Write-Host "Download fehlgeschlagen: $($_.Exception.Message)" -ForegroundColor Red
    exit 2
}
Write-Host "Download abgeschlossen." -ForegroundColor Green

# Installation starten. -NoNewWindow ist Pflicht unter SYSTEM/Session 0.
# Args werden als ARRAY uebergeben, damit PowerShell nichts re-quoten muss.
Write-Host "Starte Installation..."
$proc = Start-Process `
    -FilePath $TempSetup `
    -ArgumentList @("--proxy-url", $ProxyUrl, "--reg-secret", $RegistrationSecret) `
    -Wait -PassThru -NoNewWindow

if ($proc.ExitCode -ne 0) {
    $errMsg = if (Test-Path $ErrorLog) { Get-Content $ErrorLog -Raw } else { "Unbekannter Fehler" }
    Write-Host "Installation fehlgeschlagen (ExitCode $($proc.ExitCode)):" -ForegroundColor Red
    Write-Host $errMsg -ForegroundColor Red
    exit 1
}

# Aufraeumen
Remove-Item $TempSetup -Force -ErrorAction SilentlyContinue
Remove-Item $ErrorLog  -Force -ErrorAction SilentlyContinue
$RegistrationSecret = $null
[System.GC]::Collect()

Write-Host "=== Deployment abgeschlossen ===" -ForegroundColor Cyan
Write-Host "Der Tray-Client startet beim naechsten Benutzer-Login automatisch." -ForegroundColor Green
Was am alten Script kaputt war: kein -NoNewWindow (deadlocked unter SYSTEM/Session 0), kein TLS 1.2 (HTTPS-Reverse-Proxies haben Downloads abgewiesen), und der Error-Log wurde unter dem falschen Pfad gesucht — jeder Fehler erschien als "Unbekannter Fehler" statt mit echter Ursache.

  • Einmalig nach dem ersten Deploy eines neuen Hosts, damit Clients überhaupt eine softshelf-setup.exe zum Download haben

  • Nach Code-Änderungen am Client (UI-Fixes, neue Features, Bugfixes)
  • Nicht nötig bei Config-Änderungen (Tactical-URL, API-Key, Registration-Secret) — die sind Runtime und landen nie im Binary

Sicherheit

Dieser Abschnitt erklärt die Schutzmechanismen die der Proxy einsetzt, was sie abdecken und was nicht. Wer das System produktiv betreibt sollte ihn einmal durchlesen — danach ist klar, wo die Grenzen liegen und welche operativen Schritte zusätzlich nötig sind.

Admin-Authentifizierung & Sessions

Der Admin-Bereich (/admin/*) ist kein HTTP-Basic-Auth mehr. Alles läuft über Session-Cookies.

Lokale Logins

  • Passwort-Speicherung: Hash via hashlib.scrypt (N=16384, r=8, p=1, 16-Byte-Salt) — RFC 7914 „interactive". Klartext-Passwörter landen nirgends in der DB.
  • Vergleich: secrets.compare_digest — timing-safe gegen Timing-Side-Channels.
  • Bootstrap: Beim ersten Start des Containers wird der in der .env gesetzte Admin (ADMIN_USERNAME / ADMIN_PASSWORD) automatisch in die admin_users-Tabelle übernommen. Der scrypt-Hash wird einmalig berechnet und gespeichert. Danach ist die .env nur noch ein Notnagel: wenn alle Admin-Accounts gelöscht oder deaktiviert sind, kann sich der bootstrap-Admin einmalig wieder einloggen, und der Account wird neu provisioniert.
  • Self-Service Passwort-Änderung: Über das User-Menü oben rechts. Beim Erfolg werden alle anderen Sessions dieses Users sofort gelöscht (gestohlene Cookies sollen die Rotation nicht überleben), und ein frischer Session-Cookie wird ausgestellt.

Microsoft Entra ID SSO (optional)

Aktivierbar in den Einstellungen (siehe Abschnitt Einstellungen). Login läuft über OpenID Connect Authorization Code Flow:

  • ID-Token-Validierung: JWKS wird von Microsoft pro Tenant geholt (https://login.microsoftonline.com/<tenant>/discovery/v2.0/keys), Signatur per RS256, Issuer und Audience werden geprüft. Pflichtfelder: exp, iat, iss, sub, aud.
  • State-Token: CSRF-Schutz für den OAuth-Callback. In-Memory, single-use, 600 s TTL.
  • E-Mail-Linking: Wenn ein Microsoft-Account die gleiche E-Mail hat wie ein bestehender lokaler Admin-User, wird verlinkt — aber NUR wenn:
    • der Entra-Claim email_verified=true ist (sonst Spoofing möglich),
    • der lokale User noch keine SSO-Bindung hat (verhindert Hijack via nachträgliche E-Mail-Änderung).
  • preferred_username wird bewusst ignoriert — der Wert ist in Entra ID nicht als verifiziert markiert und kann von einem Tenant-Admin auf jeden beliebigen Wert gesetzt werden.
  • Auto-Create: Wenn aktiviert, wird beim ersten SSO-Login eines unbekannten Users automatisch ein lokaler Admin-Account angelegt. Wenn nicht: Login wird abgelehnt mit Hinweis, dass der User vorher manuell mit derselben E-Mail angelegt sein muss. Empfehlung für strenge Umgebungen: Auto-Create aus + manuelle User-Verwaltung.

Sessions

  • TTL: 12 Stunden, sliding window — jeder Request aktualisiert last_active. Inaktive Sessions laufen also nach 12 h ab, aktive bleiben gültig.
  • Speicherung: Random 32-Byte-Token in admin_sessions mit User-Agent + IP zur forensischen Nachverfolgung.
  • Cleanup: Beim Container-Start wird cleanup_expired_sessions aufgerufen, das alle abgelaufenen Einträge wegwirft.
  • Last-Active-Admin-Schutz: Der letzte aktive Admin kann weder gelöscht noch deaktiviert werden — du kannst dich nicht selbst aussperren. Beim Deaktivieren eines Users werden zusätzlich alle seine Sessions sofort weggeworfen.

Cookie-Härtung & CSRF

Session-Cookie Flags

FlagWertWarum
HttpOnlytrue JS auf der Seite kann den Cookie nicht lesen → XSS kann ihn nicht stehlen.
SameSiteStrict Browser sendet den Cookie nur bei navigation auf der eigenen Domain — keine Cross-Origin-Requests, keine GET-Bilder von fremden Sites.
Securebei HTTPS Wird automatisch gesetzt wenn der Request über HTTPS kam (direkt oder via X-Forwarded-Proto: https hinter einem Reverse-Proxy). Bei plain HTTP wird der Flag weggelassen, damit der Cookie überhaupt funktioniert.
Path/admin Nur an Admin-Endpoints gesendet. Andere Pfade kriegen den Cookie nicht.

CSRF-Schutz

State-changing Methoden (POST/PUT/PATCH/DELETE) auf /admin/api/* müssen eine der folgenden Bedingungen erfüllen:

  • Der Origin- oder Referer-Header passt zum eigenen Host (klassischer same-origin-Check).
  • ODER der Header X-Requested-With: XMLHttpRequest ist gesetzt. Browser senden den Header NIE automatisch cross-origin — das Setzen erfordert explizit einen XHR-Aufruf mit gleich-origin oder einen erlaubten CORS-Preflight, den wir nirgends erlauben.

Die Admin-UI (api()-JS-Helper) setzt beides automatisch. Cross-Origin- Aufrufe von einer fremden Webseite scheitern an der CSRF-Middleware mit 403, selbst wenn ein gestohlener Cookie mitgeschickt würde.

Rate-Limiting & Reverse-Proxy

In-Memory Per-IP-Bucket, ohne externe Dependency:

  • /api/v1/register: 5 Requests / 60 s — Brute-Force-Schutz auf das Registration-Secret. Bei erfolglosen Versuchen kommt nach dem 5. ein HTTP 429 für 60 s.
  • /admin/*: 60 Requests / 60 s — Schutz gegen Login-Brute-Force plus normaler UI-Verkehr.

X-Forwarded-For

Wenn der Proxy hinter einem Reverse-Proxy auf demselben Host (127.0.0.1 / ::1) läuft, respektiert das Rate-Limit den ersten Eintrag aus dem X-Forwarded-For-Header. Damit zählt jeder echte Client einzeln und nicht alle als „der eine Reverse-Proxy". X-Forwarded-For von externen IPs wird ignoriert (sonst trivial spoofbar).

Wichtig bei TLS-Termination: wenn du eine Reverse-Proxy-Lösung wie Caddy oder Traefik vor den softshelf-proxy stellst, muss diese den Original-Host-Header durchreichen (für CSRF) UND den X-Forwarded-For-Header (für Rate-Limit + Audit-Log). Caddy macht das per Default, nginx braucht proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; und proxy_set_header Host $host;.

Memory-Leak-Schutz

Der Bucket-Store wird alle 500 Requests gesweept: alle IPs deren Buckets seit >60 s leer sind, werden entfernt. Damit wächst der Speicher nicht unbegrenzt, auch bei vielen unterschiedlichen Client-IPs.

Machine-Tokens & Download-URLs

Machine-Token (JWT, HS256)

Jeder Kiosk-Client bekommt bei der Registrierung ein signiertes JWT, das in der Windows-Registry (HKLM\SOFTWARE\Softshelf) gespeichert wird:

  • Signaturkey: SECRET_KEY aus der .env. NIE rotieren nach Deploy — sonst werden alle Tokens ungültig und alle Kiosk-Clients müssen neu registriert werden.
  • Payload: agent_id (an Tactical-Agent gebunden), hostname (für Logging), exp (Ablauf, default 365 Tage, in den Settings änderbar; 0 = unbegrenzt), tv (Token-Version).
  • Token-Version: Wird in agents.token_version gespeichert und bei jedem Re-Register hochgezählt. Damit ist Revocation möglich: sobald die Version in der DB hochgezählt wird, werden alle bisherigen Tokens für diesen Agent ungültig — der Client muss neu registriert werden.
  • Rate-Limit: /api/v1/register ist auf 5 Versuche pro Minute limitiert.

Download-Token (signed URL)

Wenn ein Custom-Paket installiert wird, baut der Proxy eine signierte URL die den Tactical-Agent berechtigt, das File einmal herunterzuladen:

  • Bindung: Token enthält den SHA-256 der Datei, die agent_id, und eine Ablaufzeit von 5 Minuten.
  • Verifikation: Der Download-Endpoint (GET /api/v1/file/{sha256}?token=...) prüft Signatur, sha-Match und Ablauf. Andere Agents können denselben Token nicht benutzen, andere Files auch nicht.
  • Kein Auth-Header nötig: Tactical-Agent macht den Download als Invoke-WebRequest ohne Auth-Header — die Sicherheit liegt komplett im Token in der URL.

Upload- und Archiv-Härtung

File-Speicherung mit SHA-256-Dedup

  • Alle Uploads landen in /app/data/uploads/ mit dem SHA-256 als Dateinamen (<sha>.msi, <sha>.exe, <sha>.zip). Identische Uploads werden automatisch dedupliziert.
  • Beim Löschen eines Pakets/einer Version wird die physische Datei nur dann entfernt, wenn kein anderes Paket und keine andere Version denselben Hash referenziert (sha256_usage_count + count_versions_with_sha).

Path-Traversal-Schutz beim Folder-Upload

Beim Upload eines Programm-Ordners zippt der Server die Dateien serverseitig in ein einziges Archiv. Pro Datei wird der Relativpfad validiert:

  • Backslashes werden zu Forward-Slashes normalisiert.
  • ..-Segmente werden abgewiesen.
  • Leere Pfad-Segmente werden abgewiesen.
  • Absolute Pfade (/foo) werden abgewiesen.
  • Drive-Letter-Präfixe (C:) werden abgewiesen.

Beim Bearbeiten eines bestehenden Ordner-Pakets werden die Pfade aus dem Source-ZIP nochmal mit derselben Logik validiert (Defense-in-Depth, falls jemand per direktem Filesystem-Schreiben einen unsauberen Pfad ins Storage gebracht hat).

ZIP-Bomb-Defense

Bei jeder ZIP-Operation wird die dekomprimierte Größe gegen max_upload_mb (Default 500 MB) geprüft, nicht die komprimierte. Ein 10 KB ZIP, das auf 10 GB entpackt wird, schlägt also schon während des Lesens fehl.

Entry-Point-Validierung

Bei Programm-Ordner-Paketen wird der vom Admin gewählte Entry-Point gegen die tatsächlich im Archiv enthaltenen Dateien validiert (archive_entries- Cache). Damit kann der Admin keinen Pfad eintragen, der zur Laufzeit auf eine Datei außerhalb des entpackten Verzeichnisses zeigt.

Build-Pipeline-Härtung

Die softshelf.exe wird vom Builder-Container per PyInstaller unter Wine cross-kompiliert. Beim Build wird die proxy_public_url in eine _build_config.py geschrieben, die in die EXE eingebacken wird.

Vorher-Schwachstelle (gefixt): die URL wurde direkt in ein Bash-Heredoc geschrieben. Ein Admin (oder ein XSS-Opfer) hätte mit einer URL wie http://x"; import os; os.system('…') beliebigen Python-Code in jede EXE einbacken können → backdoor auf jedem Endgerät. Fix: build.sh übergibt URL+Version per ENV an Python, und Python schreibt die Werte mit repr() aus. Plus die URL wird in den Settings serverseitig auf Steuerzeichen, Quotes und Backslashes geprüft.

XSS-Hardening im Admin-UI

Die Admin-UI rendert vieles über JS-Templates mit innerHTML. Für Werte die in einem JS-String-Literal innerhalb eines HTML-Attributs landen (z.B. onclick="foo('${...}')") reicht nicht, das Wert nur HTML-zu-escapen — der HTML-Parser dekodiert &#39; zurück zu ', was dann den JS-String terminieren würde.

Die UI nutzt deshalb für solche Stellen jsStr(value), das JSON.stringify + HTML-Attribut-Encoding kombiniert. Damit sind alle Sonderzeichen in admin-controlled Daten (Display-Name, Version-Label, Hostname, etc.) sicher in inline-Handlern verwendbar.

HTML-Härtung im Kiosk-Client

Der Kiosk-Client zeigt unter dem Paketnamen Metadaten an, die teilweise von Tactical's Software-Scan kommen (Version, Publisher). Diese Daten kommen aus der Windows-Registry und sind nicht vertrauenswürdig — eine bösartige Drittanbieter-Installation könnte sie als HTML formulieren. Daher:

  • Alle interpolierten Strings werden mit html.escape() behandelt.
  • Der Display-Name läuft über QLabel.setTextFormat(Qt.PlainText).
  • setOpenExternalLinks(True) ist nur für Chocolatey-Pakete aktiv — bei Custom-Paketen werden Links nicht klickbar gemacht (verhindert file://-Schlepperei aus Tactical-Daten).

Datenbank-Integrität

  • Parametrisierte Queries: Alle SQL-Statements verwenden ?- Platzhalter. User-Input landet nie verbatim im SQL. Dynamische Query-Builder (z.B. update_admin_user) verwenden ein Whitelist-Mapping für Spaltennamen.
  • Foreign Keys: Jede aiosqlite-Connection wird über den _db()-Helper geöffnet, der PRAGMA foreign_keys = ON pro Connection setzt. Damit greifen die ON DELETE CASCADE-Klauseln im Schema (SQLite enforced FKs nicht automatisch).
  • Migrations: init_db() nutzt idempotente PRAGMA table_info-Checks plus ALTER TABLE ADD COLUMN — bestehende DBs bekommen neue Spalten ohne Datenverlust.
  • Audit-Log + Install-Log Pruning: Bei jedem Container-Start werden Einträge älter als log_retention_days (Default 90 Tage) gelöscht.

Operative Checkliste

Punkte die du operativ sicherstellen solltest, weil der Code sie nicht alleine erzwingen kann:

  1. SECRET_KEY ist lang & zufällig. 32 Bytes hex (openssl rand -hex 32). Ändert sich nach Deploy NIE — sonst werden alle Machine-Tokens ungültig und alle Kiosks müssen neu registriert werden.
  2. Bootstrap-Admin-Passwort sofort ändern. Das in der .env gesetzte Passwort wird beim ersten Start in die DB übernommen. Sobald es einen zweiten Admin gibt, kannst du den Bootstrap deaktivieren oder das Passwort im UI rotieren.
  3. Registration-Secret regelmäßig rotieren. Im Settings-Tab. Bestehende Kiosk-Clients sind davon NICHT betroffen — sie nutzen ihr Machine-Token. Nur neue Deployments brauchen das aktuelle Secret.
  4. TLS-Reverse-Proxy. Der Proxy selbst kann kein TLS — produktiv läuft er hinter Caddy/Traefik/nginx. Dann proxy_public_url auf https://... setzen, damit auch der Cookie-Secure-Flag und die Download-URLs HTTPS verwenden.
  5. Max-Upload-Größe prüfen. Default 500 MB. Wenn du große Programm- Ordner verteilst, hochsetzen. Aber sehr großzügig sein heißt auch: ein boshafter Admin könnte Speicher voll-uploaden.
  6. Admin-Accounts minimal halten. Jeder Admin kann beliebige PowerShell- Commands an alle Kiosk-Clients schicken (über uninstall_cmd / install_args / Custom-Pakete). Das ist ein SYSTEM-RCE auf jedem Endgerät. Die Admin-Rolle ist effektiv Domain-Admin-Niveau.
  7. SSO Auto-Create nur in vertrauenswürdigen Tenants. Wenn aktiviert, wird jeder Tenant-User mit Login-Pfad zum Self-Service Center automatisch zum Admin. In Multi-Tenant-Szenarien NIE aktivieren — manuell User anlegen.
  8. Backup der DB. /opt/softshelf/data/softshelf.db. Da drin sind Admin-Hashes, Sessions, Tracking, alles. Ein DB-Verlust = alle Kiosks verlieren ihren Auth-Status nicht (das Token ist client-side), aber der Admin-Zugang muss neu provisioniert werden.
  9. Logs anschauen. docker logs softshelf-proxy zeigt alle Auth-Versuche, Build-Fehler und Background-Task-Crashes. Audit-Log im UI zeigt alle HTTP-Requests der letzten 200 Treffer.

Was wir bewusst nicht tun

  • Keine Multi-Faktor-Authentifizierung im lokalen Login. Wer MFA will, aktiviert SSO mit Microsoft Entra ID — dort gibt es MFA, Conditional Access, Geo-Filter etc.
  • Keine Per-Action-Approval. Ein Admin allein kann jedes Custom-Paket auf jeden Client pushen. Wer 4-Augen-Prinzip braucht, muss einen separaten Approval-Schritt einführen (gibt's heute nicht).
  • Keine Crypto-signierten Pakete. Wir vertrauen darauf dass der Admin saubere MSI/EXE hochlädt. Tactical-Agent installiert was auch immer, ohne Code-Signing-Verifikation.
  • Keine API-Keys / Service-Accounts. Es gibt nur Admin-User mit interaktiven Sessions. Wer Automatisierung braucht, muss das Session-Cookie aus einem Login-Call extrahieren und für die Folge-Calls verwenden — nicht schön, aber funktioniert.

Auf neuen Host migrieren

Ziel: das Self-Service Center auf eine neue Linux-Maschine umziehen, ohne Datenverlust und mit möglichst wenig Client-seitigen Anpassungen.

Variante A: Komplett frisch (ohne Daten-Übernahme)

  1. Repo auf neuem Host klonen: git clone … /opt/softshelf
  2. cp .env.example .env → neues SECRET_KEY generieren (openssl rand -hex 32), ADMIN_USERNAME und ADMIN_PASSWORD setzen
  3. docker-compose up -d --build (erster Build ~5-8 min)
  4. Im Admin-UI Einstellungen ausfüllen (Tactical-URL, API-Key, Registration-Secret, Proxy-URL für den neuen Host)
  5. EXEs bauen
  6. Neue softshelf-setup.exe an alle Clients ausrollen (Tactical-Script oder manuell)
Da SECRET_KEY neu generiert wird, sind alle alten Machine-Tokens ungültig. Alle Clients müssen neu registriert werden.

Variante B: Mit Daten-Übernahme (alten Zustand behalten)

  1. Auf dem alten Host stoppen: docker-compose down
  2. Die data/-Verzeichnis und die .env sichern: 
    tar czf softshelf-backup.tgz data .env
  3. Archiv auf den neuen Host transferieren (scp, USB, etc.)
  4. Auf dem neuen Host das Repo klonen: git clone … /opt/softshelf
  5. Backup extrahieren: tar xzf softshelf-backup.tgz -C /opt/softshelf/
  6. docker-compose up -d --build
  7. Im Admin-UI Einstellungen prüfen — die migrieren sich aus der SQLite. Proxy Public URL wahrscheinlich aktualisieren auf den neuen Host.
  8. EXEs bauen (damit die eingebackene URL stimmt)
  9. Clients neu ausrollen (siehe oben)

Vorteil Variante B: Whitelist, Audit-Log, Install-Log, custom-Pakete und alle Einstellungen sind direkt da. Nur die Clients müssen erneut deployt werden, wenn sich die Proxy-URL ändert.

Tipp: Wenn du auf dem neuen Host denselben DNS-Namen wie auf dem alten nutzt (z. B. softshelf.intern als CNAME auf die neue IP), dann brauchen die Clients überhaupt kein Re-Deployment — sie erreichen den neuen Proxy unter derselben URL und benutzen ihre bestehenden Machine-Tokens (vorausgesetzt du hast SECRET_KEY und data/ übernommen).

Troubleshooting

Kiosk zeigt "Verbindungsfehler: Token wurde widerrufen"

Ursache: Die token_version im JWT stimmt nicht mehr mit der in der DB überein. Das passiert wenn:

  • Das Gerät zwischenzeitlich über Tactical-Script re-deployt wurde (bumpt token_version)
  • Ein Admin manuell per SQL die token_version hochgezählt hat
  • Nach einem Proxy-SECRET_KEY-Wechsel (der seltene Fall)

Lösung: Client erneut deployen (Tactical-Script oder manueller Install). Dabei wird ein neues Token erzeugt mit aktueller tv.

Kiosk zeigt "Token abgelaufen"

Ursache: Der exp-Claim des JWT ist vorbei. Standard ist 365 Tage nach Registrierung.

Lösungen:

  • Client erneut deployen (selber Vorgang wie Erstinstall, überschreibt das Token)
  • In den Einstellungen token_ttl_days=0 setzen — damit laufen zukünftige Tokens nie ab. Bestehende Tokens mit exp sind trotzdem weiterhin abgelaufen, für die brauchst du einen Re-Deploy.

Paket wurde installiert, aber Kiosk zeigt "Nicht installiert"

Ursache 1: Tactical's Software-Scan ist noch nicht durchgelaufen. Die Detection basiert auf Tactical's Liste der installierten Programme — die wird periodisch aktualisiert, nicht in Echtzeit.

Lösung: Warten (Minuten bis zur nächsten Software-Scan-Runde) oder in Tactical manuell "Scan Installed Software" auf dem Agent triggern.

Ursache 2: Der detection_name des Pakets stimmt nicht mit dem Windows-Anzeigenamen überein.

Lösung:

  1. Im Kiosk-Client-Admin: Edit-Panel des Pakets öffnen
  2. In Tactical unter „Software" des Agents nachschauen wie das Programm genau heißt
  3. Detection-Name entsprechend anpassen (Substring reicht — z. B. nur "OpenVPN" statt "OpenVPN 2.7.1-I001 amd64")
  4. Speichern

Upload schlägt fehl

Symptom: Upload-Button zeigt Fehlermeldung, Paket erscheint nicht in der Liste.

Mögliche Ursachen:

  • Datei zu groß: default 500 MB. Ändern via Einstellung max_upload_mb.
  • Ungültige Extension: nur .msi und .exe erlaubt.
  • Ungültiger Anzeigename: nur druckbare Zeichen, max 80 Zeichen.
  • Paketname (Slug) bereits vergeben: der Slug wird aus dem Dateinamen abgeleitet; bei Konflikt wird automatisch ein Suffix angehängt. Sollte nicht fehlschlagen.

Genaue Ursache im Admin-UI Tab Audit-Log, Zeile des POST /admin/api/upload mit Status-Code checken. Bei 500: im Proxy-Container-Log schauen.

"EXEs bauen" schlägt fehl

Tab Einstellungen → unter Build-Historie den letzten Build mit Status failed finden → Log klicken → vollständiger PyInstaller-Output.

Typische Fehler:

  • Proxy Public URL nicht gesetzt: der Build wird sofort abgelehnt. Einstellung ausfüllen.
  • Wine-Crash in Python: passiert sporadisch mit bestimmten Python-Imports unter Wine. Meist hilft ein zweiter Build-Versuch.
  • Client-Code-Syntax-Fehler: wenn kürzlich Änderungen am client/-Verzeichnis gemacht wurden, kann ein Syntax-Fehler im PyInstaller-Analyze-Step auftauchen. Fehlermeldung im Log enthält den exakten Fehler + Datei/Zeile.

Tactical-API 502 (Proxy-seitiger Fehler)

Symptom: Kiosk-Client zeigt "Fehler: HTTP 502" beim Laden der Paketliste oder beim Install.

Ursachen & Lösungen:

  • Tactical nicht erreichbar: Netzwerkproblem oder Tactical-Service down. Per curl vom Proxy-Host aus die Tactical-URL anpingen.
  • Falscher API-Key: Tab Einstellungen → tactical_api_key → Reveal + prüfen.
  • API-Key hat nicht genügend Permissions: In Tactical unter User-Management dem API-Key-User die Permissions can_manage_software und can_send_cmd geben (siehe Referenz unten).
  • Agent gelöscht: Wenn ein Agent in Tactical gelöscht wurde, aber der Kiosk-Client seinen Token noch hat → 404 von Tactical → 502 im Kiosk. Lösung: Agent wieder in Tactical registrieren oder Kiosk-Client deinstallieren.

Admin-Login geht nicht mehr

  • Prüfen ob das Passwort via Runtime-Setting admin_password_override geändert wurde. Falls ja: falls vergessen → direkt in der SQLite-DB löschen: 
    docker exec softshelf-proxy sqlite3 /app/data/softshelf.db \
  "DELETE FROM settings WHERE key='admin_password_override'"
    Danach greift wieder das Bootstrap-Passwort aus der .env.
  • Wenn auch das .env-Passwort nicht mehr bekannt ist: .env editieren, docker-compose restart softshelf-proxy.

Rate-Limit (429 Too Many Requests)

Passiert bei >5 Registration-Versuchen pro Minute oder >60 Admin-Calls pro Minute von derselben IP. Einfach einen Moment warten. Bei absichtlichen Load-Tests das Rate-Limit in proxy/middleware/rate_limit.py anpassen (Code-Change + Restart).

Tactical-User-Permissions

Der Tactical-API-Key, der in tactical_api_key hinterlegt ist, muss einem Tactical-User gehören, der mindestens folgende Permissions hat:

  • can_list_software — GET auf Software-Endpoints für die Installations-Detection
  • can_manage_software — POST /software/{id}/ (Chocolatey-Install) und POST /software/{id}/uninstall/
  • can_send_cmd — POST /agents/{id}/cmd/ (für Custom-MSI/EXE-Install, Choco-Uninstall mit explizitem Command)
  • Agent-Zugriff auf die Clients, die das Self-Service Center nutzen sollen (über Role-assigned Clients/Sites)

Permission fehlt → Tactical antwortet mit HTTP 403 → Proxy gibt HTTP 502 an den Client zurück.
