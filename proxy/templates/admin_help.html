Schnellstart

Die fünf häufigsten Aktionen:

📦

Paket freigeben

Tab Pakete → + Paket hinzufügen → winget/choco/custom wählen → Name suchen → +.

🚀

Staged Rollout starten

Tab Pakete → ▶-Icon bei staged Paket. Phase 1 (Canary) geht sofort los, danach im Rollouts-Tab weiter.

⚠

Fehler abarbeiten

Tab Fehler zeigt offene Install-Fehler flottenweit. Erneut oder ✓ zum Bestätigen.

✅

Required-Compliance prüfen

Tab Compliance: Required-Pakete mit Abdeckung pro Agent. „Alles nachinstallieren" für einen Rutsch.

⏰

Wartungsfenster anlegen

Tab Geplant → + Neuer Job → Zeitpunkt + Aktion + Stage wählen. Läuft automatisch zur eingestellten Zeit.

🔍

Quick-Jump

⌘K / Strg+K öffnet Blitzsuche über alle Tabs, Pakete, Agents und Profile.

Rollen & Rechte (RBAC)

{{SF_BRAND}} kennt drei Rollen. Eingestellt pro Admin-User unter Benutzer.

RolleDarfDarf nicht
admin Alles: Whitelist, Benutzer, Einstellungen, Build, Dispatch, Paket-Flags, Agent-Ringe —
operator Dispatch (Install/Uninstall/Update-All), Push-Update, Bulk-Aktionen, Rollouts starten & weiterschalten, Profile anwenden, Fehler bestätigen Whitelist-Edit, Users anlegen, Settings ändern, Build triggern, Required/Staged-Flags setzen, Ring umkonfigurieren
viewer Alle GETs (lesen, durchklicken, CSV-Export) Jegliche state-changing Action → 403
Durchsetzung: Der Server ist Source of Truth. Die UI versteckt oder disabled Admin-Only Controls für andere Rollen kosmetisch, aber jeder API-Call wird backend-seitig in _require_admin gegen method+path geprüft.

Paket-Typen

{{SF_BRAND}} kennt vier Paket-Quellen. Type-Pills in der UI zeigen den Typ mit farbigem Dot:

TypQuelleDispatchBesonderheit
winget Microsoft-winget-Catalog, lokal gemirrowt; plus MS-Store via Direct-ID winget install/upgrade/uninstall via Tactical run_command Scope (auto/machine/user) pro Paket — Layer-2-Fallback für per-user-only-Installer wie LastPass, Bitwarden. MS-Store-IDs (z.B. 9NBLGGH6BZL3) werden auto erkannt und mit --source msstore + run_as_user installiert.

custom Manueller MSI/EXE/Ordner-Upload Download via signed-URL → msiexec/Start-Process Version-Tracking, Push-Updates.
plugin Plugin-Datei für eine Host-Anwendung (Notepad++/KeePass) PowerShell kopiert die Datei in den Plugin-Ordner der Host-App Auto-Process-Check (Host-App muss zu sein), Soft-Error wenn Host-App nicht installiert. Update via Re-Upload (sha256-Detection).

Winget-Paket freigeben

  1. Tab Pakete → + Paket hinzufügen.
  2. Reiter Winget. Tippen → Live-Suche im lokalen Catalog-Mirror.
  3. Plus-Button bei Treffer → aktiviert das Paket in der Whitelist.
  4. Optional: Scope inline einstellen:
    • auto (Default): erst SYSTEM + --scope machine, bei „No applicable installer" auto-retry im User-Kontext.
    • machine: nur SYSTEM, kein Fallback.
    • user: direkt User-Kontext — für LastPass, Bitwarden, Firefox-per-user.
Discovery: Das Panel zeigt außerdem „In der Flotte gefunden" — winget-Pakete die auf euren Clients bereits installiert sind aber noch nicht in der Whitelist. One-click-Aktivieren möglich.

Choco-Paket freigeben

  1. Tab Pakete → + Paket hinzufügen → Reiter Aus Chocolatey.
  2. Name tippen, Plus drücken.
FOSS-Hinweis: Manche choco-Pakete (die mit .install-Suffix) laden Installer vom Hersteller-CDN, nicht von Chocolatey selbst. Bei 404 oder „Likely broken for FOSS users" steht die Fehlermeldung mit Link zum Paket-Log im Fehler-Tab.

Eigenes MSI/EXE/Ordner hochladen

  1. Tab Pakete → + Paket hinzufügen → Eigene Datei / Programm-Ordner.
  2. Datei oder Ordner wählen. MSI-Metadaten werden automatisch extrahiert.
  3. Optional: Silent-Install-Args anpassen. Default für MSI: /qn /norestart.
  4. Uninstall-Command: via Detect-Button aus Tactical übernehmen oder manuell hinterlegen.
  5. Hochladen.

Nachträgliches Bearbeiten: Stift-Icon in der Paket-Zeile → Panel mit allen Feldern, inklusive Versions-History und „Inhalt verwalten" bei Ordner-Paketen.

Bulk-Import winget-IDs

Wenn du 50 Pakete auf einmal aktivieren willst:

  1. Tab Pakete → Bulk-Import.
  2. Textarea: eine winget-ID pro Zeile (oder komma-getrennt).
  3. Kategorie setzen.
  4. Importieren. Metadaten (Name, Publisher) werden automatisch aus dem Catalog geholt. Bereits aktive Pakete werden übersprungen, ungültige IDs landen in der Fehler-Liste.

Plugin hochladen (Notepad++ / KeePass)

Plugins für Anwendungen mit File-basierter Plugin-Architektur — Drop-in-DLL/PLGX, kein Installer. Aktuell unterstützt:

  • Notepad++ — .zip oder .dll. Datei landet unter C:\Program Files\Notepad++\plugins\<PluginFolder>\.
  • KeePass 2 — .plgx oder .dll. Datei landet flach unter C:\Program Files\KeePass Password Safe 2\Plugins\.
  1. Tab Pakete → + Paket hinzufügen → Gruppe Hochladen → Reiter Plugin.
  2. Plugin-Host wählen (Notepad++ / KeePass 2). Die akzeptierten Datei-Endungen passen sich automatisch an.
  3. Plugin-Datei auswählen. Magic-Bytes werden geprüft (Spoofing-Schutz).
  4. Anzeigename setzen (z.B. „ComparePlus").
  5. Plugin-Ordner / Dateiname — leer lassen für Auto-Ableitung aus Dateinamen, oder explizit setzen wenn der Ordner anders heißen soll.
  6. Kategorie leer lassen → Auto-Setzung auf Host-Label (z.B. „Notepad++"), damit das Plugin im Kiosk unter seiner Host-App gruppiert wird.
  7. Hochladen.
Update: Re-Upload mit anderem Datei-Inhalt = neuer sha256 → alle Agents mit altem Hash gelten als outdated. Im Edit-Panel (Stift) erscheint dann der Push-Update-Button mit Anzahl betroffener Geräte.
Vor Install muss Host-App zu sein. NPP/KeePass-Prozess wird automatisch erkannt — wenn er läuft, sieht der User einen Modal „Anwendung schließen und nochmal versuchen". Sonst: Locked-DLL-Error.
Wenn die Host-App fehlt (z.B. Plugin auf einem Rechner ohne Notepad++ installiert), wird der Install mit Soft-Error 9030 abgebrochen — kein Müll im Filesystem, klare User-Fehlermeldung.

MS-Store-App per ID hinzufügen

Manche Apps gibt es nur im Microsoft Store (z.B. GlobalProtect, Dell Command Update). Diese kann winget mit --source msstore installieren — {{SF_BRAND}} erkennt das automatisch am ID-Format.

  1. Auf einem Test-Agent in PowerShell winget search <name> --source msstore ausführen. Die ID beginnt mit 9 und ist 12 Zeichen lang (z.B. 9NBLGGH6BZL3).
  2. Tab Pakete → + Paket hinzufügen → Winget.
  3. Klick auf den kleinen Hinweis-Link „MS-Store-Apps per ID hinzufuegen →" unter dem Suchfeld.
  4. Modal ausfüllen: Store-ID, Anzeigename, Publisher (optional), Kategorie.
  5. Hinzufuegen.
Was passiert beim Install: {{SF_BRAND}} erkennt das ID-Format und setzt automatisch --source msstore und run_as_user=True (MS-Store-Apps sind Per-User-Appx). Der Agent muss einen eingeloggten User haben.

Paket-Optionen (Edit-Panel)

Im Edit-Panel jedes Pakets (Stift-Icon in der Paket-Liste) gibt es Optionen, die das Verhalten beim Install / Anzeige im Kiosk steuern:

FeldWirkung
Install-ArgsZusätzliche CLI-Argumente, die hinter winget install / choco install drangehängt werden. Beispiele winget: --scope user, --silent, --accept-package-agreements. Choco: --params "/NoDesktopIcon".
Process-CheckKomma-separierte Prozessnamen (mit oder ohne .exe). Wenn beim Install einer dieser Prozesse läuft, bricht die Installation mit klarer Fehlermeldung ab. Der Kiosk zeigt dem User „Bitte X schließen" als Modal. Beispiel: firefox,firefox.exe.
Uninstall ausblenden (hide_uninstall)Wenn aktiviert, sehen End-User im Kiosk keinen „Deinstallieren"-Button für dieses Paket. Admin kann weiter über die Verteilung-Sicht deinstallieren. Typisch für Required-Pakete (AV, Backup-Agent).

1 = versteckt: Paket erscheint nie im Kiosk, nur Admin sieht es. Trotzdem dispatchbar über Profile/Workflows.
2 = nur wenn installiert: Paket erscheint im Kiosk-Tab „Installiert", aber nicht in „Alle". Sinnvoll für Pakete, die nicht freiwillig gewählt werden sollen — der User sieht nur dass es da ist und kann updaten.
Install-TimeoutSekunden, die der Installer maximal laufen darf. Default 120s. Office/SAP: 600–900s.

Versionen-History (Custom-Pakete)

Custom-Pakete (eigene MSI/EXE/Ordner) haben einen vollständigen Versions-Verlauf. Bei jedem Upload einer neuen Version wird die alte als archived markiert; die neue wird current und ist die, die Agents installieren.

  1. Tab Pakete → Custom-Paket öffnen (Stift-Icon).
  2. Im Edit-Panel: Versionen-Tab (rechts oben im Panel).
  3. Liste aller Uploads chronologisch mit Datum, Größe, Uploader, sha256.

Pro Version verfügbar:

  • Als aktuell setzen — alte Version wird wieder die aktive. Push-Update kann sie nun an alle Agents verteilen.
  • Beschreibung editieren — Versions-Label / Notiz für interne Doku.
  • Löschen — Datei + DB-Eintrag entfernt. Nicht möglich wenn die Version gerade current ist.
Versionen-History gibt es nur für custom-Pakete. winget/choco managen ihre Versionen über den jeweiligen Catalog, plugins haben sha256-Tracking pro Agent ohne expliziten History-Tab.

Version Pinning

Winget- und Choco-Pakete werden standardmäßig auf latest gehalten — beim nächsten Upgrade wird die neueste verfügbare Version installiert. Mit einem Version Pin kannst du ein Paket auf eine bestimmte Version fixieren.

Pin setzen

  1. Tab Pakete → Paket anklicken (winget oder choco) → Bearbeiten.
  2. Im Bearbeitungs-Panel erscheint ein Version-Dropdown. Klick auf das Feld öffnet die Versions-Liste:
    • Winget: Versionen aus dem lokalen Catalog-Mirror.
    • Choco: Versionen die der Fleet-Scan auf mindestens einem Agent gesehen hat.
  3. Version auswählen oder manuell eingeben → Speichern.

Der gesetzte Pin erscheint als Label v2.3.1 in der Paket-Liste. Speichern ohne Auswahl setzt den Pin zurück auf latest.

Verhalten bei gesetztem Pin

  • Neuinstallation: winget/choco installiert exakt die gepinnte Version (--version <pin>).
  • Upgrade: Upgrade wird nur ausgeführt wenn die installierte Version ≠ Pin. Ist die gepinnte Version schon drauf, wird der Upgrade-Schritt übersprungen.
  • Update-All und Profile-Auto-Update berücksichtigen den Pin ebenfalls.
Wichtig: Version-Pin gilt fleet-weit für das Paket. Für abweichende Versionen pro Team nutze Profile — dort gibt es ebenfalls einen version_pin pro Profil-Eintrag.
Choco: Die Versionen-Liste kommt aus dem Fleet-Scan (agent_choco_state), nicht direkt aus der Chocolatey-Registry. Wenn keine Agents das Paket installiert haben, ist die Liste leer — du kannst die Version trotzdem manuell eintippen.

Profile

Profile sind Bundles aus mehreren Paketen, zuweisbar auf beliebige Clients.

  1. Tab Profile → + Neues Profil → Name, Beschreibung, Farbe.
  2. Paket-Picker: welche Pakete enthält das Profil. Optional: Version-Pin pro Paket.
  3. Speichern → Profil erscheint in der Liste links.
  4. Client zuweisen: Profil auswählen → Clients zuweisen → Agent-Picker.
  5. Apply löst sofort Install für alle nicht-vorhandenen Pakete aus.
Auto-Update: Option im Profil-Editor. Wenn gesetzt, pusht der nightly Job (03:00 UTC) Updates für alle Pakete im Profil an die zugewiesenen Agents. Smart-Skip: bereits aktuelle Installs werden übersprungen.
Propagation: Fügst du ein Paket zu einem Profil hinzu, wird es sofort auf allen zugewiesenen Clients installiert (außer sie haben's schon). Entfernen ohne Uninstall — der Admin muss explizit „Unassign mit Uninstall" wählen um zu deinstallieren.

Phased Rollouts

Konzept: Ringe, Phasen und Zuordnung

Zwei Dinge klar trennen

Ein Staged Rollout beantwortet nur eine Frage: wann bekommt ein Client das Update? Ob ein Paket überhaupt zu einem Client gehört, entscheidet eine andere Mechanik — die Zuordnung.

MechanikAntwortet aufWird gesteuert durch
Zuordnung Ist dieses Paket auf diesem Client installiert (oder sollte es sein)? Profil-Zuweisung · Required-Flag · direkter Admin-Push · User-Install im Kiosk
Rollout-Phase Wann geht das Update zu welchem Ring? Staged-Flag + Ring pro Agent + aktive Rollout-Phase
Häufiger Irrtum: Staged-Flag aktivieren reicht nicht aus damit ein Paket auf neue Clients installiert wird. Staged steuert nur den Rollout bestehender Zuweisungen. Ein Client der das Paket gar nicht hat, bekommt es auch bei erreichter Phase nicht — egal wie weit der Rollout ist.

Konkrete Beispiele

AusgangslageAdmin-AktionErgebnis auf dem Client
Firefox installiert auf Client A (Ring 1). Staged aktiv, Rollout Phase 1 Neue Version 128 verfügbar ✓ Client A bekommt Update (Ring 1 ≤ Phase 1)
Firefox installiert auf Client B (Ring 3). Rollout Phase 1 — ✗ Kein Update-Button im Kiosk (Ring 3 > Phase 1). Wartet auf Phase 3
Firefox nicht auf Client C installiert. Staged + Phase 3 — ✗ Firefox erscheint nicht auf Client C — fehlende Zuordnung. Rollout ignoriert ihn komplett
Firefox in Profil „Devs" enthalten, Client C hat das Profil. Staged + Phase 1 Profil-Apply bei Zuweisung Firefox wird auf Client C installiert (Profil-Apply). Updates respektieren dann Ring/Phase

Ringe konfigurieren

Jeder Agent hat genau einen Ring: Ring 1 (Canary), Ring 2 (Pilot), oder Ring 3 (Produktion = Default für neu registrierte Agents). Setzbar im Agent-Detail-Header (admin-only). Labels pro Umgebung umbenennbar unter Rollouts → Einstellungen.

Phasen-Mapping

Eine Rollout-Phase löst an einen Ring aus:

  • Phase 1 → alle Agents mit Ring 1
  • Phase 2 → alle Agents mit Ring 2
  • Phase 3 → alle Agents mit Ring 3 (Produktion)

Durch Kiosk-Gate sehen Agents in späteren Ringen das Update erst wenn ihre Phase erreicht ist — auch bei User-initiierten Update-Checks im Kiosk.

Pakete mit staged_rollout=1 (Haken-Icon in Pakete-Row) müssen diese Phasen durchlaufen:

Phase 1
Ring 1 (Canary)
kleinste Testgruppe
→
Phase 2
Ring 2 (Pilot)
größere Testgruppe
→
Phase 3
Produktion
alle Nicht-Test-Agents

Zwischen den Phasen prüft der Admin (oder der Auto-Modus) ob Fehler auftraten. Bei Problemen: Rollout abbrechen oder pausieren.

Workflow: Paket staged ausrollen

  1. Paket als phased markieren: Tab Pakete → ✓-Icon pro Zeile klicken (aktiv = amber).
  2. Agent-Ringe zuweisen: Tab Clients → Agent öffnen → Ring-Dropdown im Header → 1 für Test-Gerät setzen.
  3. Rollout starten: Tab Pakete → ▶-Icon bei phased Paket mit Updates. Bestätigen → Phase 1 wird sofort ausgelöst.
  4. Status prüfen: Tab Rollouts zeigt aktive Rollouts. Oder auf Übersicht.
  5. Fehler checken: Tab Fehler nach Problemen durchsehen.
  6. Weiter: Weiter-Button im Rollouts-Tab → Phase 2 (Ring 2).
  7. Nochmal prüfen, dann Weiter → Phase 3 (Produktion).
  8. Nach Phase 3 ist Rollout automatisch done.
Abbruch: Abbrechen-Button stoppt den Rollout. Bereits dispatcht Phasen bleiben auf den Agents — Rollback ist ein separater Uninstall (Profil-Unassign mit Uninstall oder manuelles choco/winget uninstall).

Phase ist Zeit, nicht Agent (seit v2.4.1)

Ein Auto-Rollout startet Phase 1 sobald irgendwo im Fleet outdated Agents existieren — egal welcher Ring. Leere Phasen laufen no-op durch und werden vom Advance-Tick nach Ablauf der Wartezeit weitergeschaltet. So sieht der Admin den Phasen-Fortschritt durchgehend, auch wenn z.B. Ring 1 (Canary) noch leer ist.

Detailansicht pro Paket

Im Rollouts-Tab pro Zeile auf ▼ Details klicken. Zeigt:

  • Agents pro Ring gruppiert mit Counts (Ring 1/2/3 separate Sektionen)
  • Status-Pill pro Agent: ✓ aktuell, outdated, ⟳ läuft, ⚠ Fehler, nicht installiert
  • Versions-Diff: installed → target bei outdated Agents
  • Rollout-Historie mit Versions-Badge → <version> pro Eintrag
  • Klick auf Hostname springt zur Agent-Detailansicht

target_version eingefroren beim Start

Beim Anlegen eines Rollouts wird die Ziel-Version festgeschrieben (rollouts.target_version). Auch wenn der winget-Catalog danach eine neuere Version liefert, zeigt die Historie weiterhin was damals tatsächlich rolled out wurde. Outdated-Vergleich pro Agent nutzt seit v2.4.1 semver-Compare (toleriert "1.2" vs "1.2.0" und Locale-Suffixe wie "11.0.27.0 (de_de)").

Workflows

Workflows sind wiederverwendbare mehrstufige Automations-Sequenzen, die auf einzelnen Agents ausgeführt werden. Typische Anwendungsfälle: komplexe Software-Setups mit Zwischenschritten, erzwungene Neustarts nach Updates, benutzerdefinierte Konfigurations-Skripte.

Tab Workflows (Sidebar-Gruppe Verwaltung) für alle Workflow-Definitionen. Workflow starten: Tab Clients → Agent anklicken → Abschnitt Workflows → Starten.

Step-Typen

TypBeschreibungParameter
install Installiert ein {{SF_BRAND}}-Paket auf dem Agent (gleicher Dispatch-Pfad wie manuell aus der Paket-Liste — mit Output-Capture und Soft-Error-Detection). Paket auswählen (Dropdown). Optional: version_pin für diesen Step überschreiben.
script Führt ein eingebettetes PowerShell-Script auf dem Agent aus. Läuft als SYSTEM über Tactical run_command. Exit-Code 0 = Erfolg. Freitextfeld für PS-Code. Code wird als temporäre .ps1-Datei geschrieben und mit -File aufgerufen, damit exit N den Callback-Wrapper nicht abbricht.
reboot Initiiert einen Neustart. Zeigt dem angemeldeten User einen Dialog mit Countdown. Nach force_after_hours wird der Neustart erzwungen, auch ohne User-Reaktion. force_after_hours (Default 8h): Zeit bis zum Zwangs-Neustart. Ein AtStartup-Scheduled-Task sendet nach dem Hochfahren den Erfolgs-Callback.

Reboot-Step im Detail

Der Reboot-Step ist der komplexeste Step-Typ, weil er Client-Interaktion erfordert:

  1. Die Engine registriert einen AtStartup-Scheduled-Task auf dem Agent. Dieser Task sendet nach dem nächsten erfolgreich abgeschlossenen Boot einen Callback an den Proxy (bestätigt dass der Neustart stattgefunden hat).
  2. Gleichzeitig wird ein Shutdown-Countdown eingeleitet (shutdown /r /t 300 = 5 Minuten Vorlauf).
  3. Der Kiosk-Client pollt regelmäßig GET /api/v1/health und erhält dabei pending_actions. Die Tray-App zeigt automatisch den Reboot-Dialog an: Countdown-Anzeige + Fortschrittsbalken.
  4. „Jetzt neu starten": startet sofort (shutdown /r /t 0).
  5. „Verschieben": schließt den Dialog für diese Runde. Der Countdown läuft weiter; bei nächstem Health-Poll erscheint der Dialog wieder.
  6. Nach Ablauf von force_after_hours: der APScheduler-Job _workflow_timeout_check erkennt die abgelaufene Deadline und triggert einen Zwangs-Neustart via Tactical (shutdown /r /t 60). Der Run bleibt danach in einem Wartezustand bis der AtStartup-Callback eingeht.
Der Reboot-Dialog erscheint nur wenn der Kiosk-Client läuft und ein User eingeloggt ist. Ohne laufenden Client kann der User den Dialog nicht sehen — der Zwangs-Neustart nach force_after_hours ist dann das einzige Sicherheitsnetz.

Failure-Policies

Jeder Step hat eine konfigurierbare on_failure-Policy:

PolicyVerhalten bei Step-Fehler
abort (Standard)Run wird sofort mit status=failed beendet. Keine weiteren Steps werden ausgeführt.
skipFehler wird ignoriert, der nächste Step wird ausgeführt. Nützlich für optionale Schritte.
retry:NBis zu N Wiederholungsversuche. Nach dem N-ten Fehlversuch: abort.

Workflow starten & beobachten

Workflow anlegen

  1. Tab Workflows → + Neuer Workflow.
  2. Name + optionale Beschreibung eingeben.
  3. Steps per Drag & Drop sortieren. Typ und Parameter pro Step konfigurieren.
  4. Speichern.

Workflow auf Agent starten

  1. Tab Clients → Agent anklicken → Detail-Seite öffnet sich.
  2. Abschnitt Workflows → Workflow aus Dropdown auswählen → Starten.
  3. Der Run erscheint sofort in der Run-Liste mit Status und aktuellem Step.
Pro Agent kann immer nur ein Workflow-Run aktiv sein. Ein zweiter Start-Versuch solange ein Run läuft gibt einen Fehler — erst den laufenden Run abschließen oder abbrechen.

Run-Status

StatusBedeutung
runningAktiver Run, wartet auf Step-Abschluss.
completedAlle Steps erfolgreich abgeschlossen.
failedEin Step ist mit abort-Policy fehlgeschlagen.
cancelledManuell abgebrochen.
timed_outStep-Deadline abgelaufen (kein Callback vom Agent eingegangen).

Run abbrechen / pausieren / fortsetzen

  • Abbrechen — Status cancelled, kein nächster Step. Bereits laufende Step-Dispatches auf dem Agent laufen zu Ende.
  • Pausieren — Status paused. Engine wartet, kein Auto-Advance. Step-Deadline wird ebenfalls eingefroren.
  • Fortsetzen — von paused zurück zu running. Neue Step-Deadline wird gesetzt; bei Reboot-Steps wird der AtStartup-Task neu registriert wenn der Original-Task weg ist.

Buttons im Workflow-Run-Card (Agent-Detail-Seite). Pause/Resume sind kein Ersatz für Abbrechen — sie verhindern nur den nächsten Tick.

Workflow im Kiosk freigeben (v2.6.0)

End-User können freigegebene Workflows direkt aus dem Tray-Kiosk starten — kein Admin-Eingriff nötig. Vorgehen:

  1. Tab Workflows → Workflow öffnen → Checkbox Im Kiosk anbieten aktivieren.
  2. Optional: Textfeld Beschreibung für Kiosk ausfüllen — der Text erscheint im Confirm-Dialog beim End-User. Wenn leer, wird die normale Workflow-Beschreibung benutzt.
  3. Speichern. In der Workflow-Liste erscheint jetzt ein KIOSK-Badge.
Sobald freigegeben ist der Workflow auf jedem Tray-Kiosk sichtbar — eine zusätzliche Agent-Zuweisung ist nicht nötig. Wer den Workflow ausführt wird mitgeloggt (workflow_runs.created_by = "kiosk:<windows-user>").

Im Kiosk taucht ein zusätzlicher Tab Aktionen auf (nur sichtbar wenn mindestens ein Workflow verfügbar ist). User klickt Ausführen → Confirm-Dialog mit Beschreibung + Step-Liste → Starten. Während der Run läuft zeigt die Karte Progress-Bar + aktuellen Schritt; Reboot-Steps öffnen weiterhin den bekannten Reboot-Dialog.

Rate-Limit: 5 Starts pro 60 Sekunden pro Agent-IP. Pro Agent darf weiterhin nur ein Workflow gleichzeitig laufen.

Offline-Queue (v2.6.x)

Wenn ein Agent ueber Tactical RMM gerade nicht erreichbar ist, scheitern Install/Uninstall/Tray-Update nicht mehr hart, sondern landen als queued (gelber Status) im Aktionen-Tab.

  • Ein APScheduler-Job (_queued_actions_tick) prueft alle 2 Minuten den Agent-Status und dispatched den Befehl sobald der Agent online ist.
  • Bei einzelnen Install/Uninstall-Klicks fragt das UI vor dem Vormerken („Agent offline — vormerken?"). Bei Bulk-Installs und Profile-Apply wird automatisch gequeued (kein Modal pro Agent).
  • Tray-Update via Queue nimmt immer die aktuelle Build-Version zum Dispatch-Zeitpunkt — nicht die Version, die zum Vormerk-Zeitpunkt aktiv war. Wenn zwischen Vormerken und Dispatch ein neuer Build entsteht, bekommt der Agent den neuen Build.
  • Aufraeumung: queued Eintraege aelter als 7 Tage oder mit nicht mehr existierendem Agent werden automatisch auf cancelled gesetzt.
  • Manueller Abbruch: Aktionen-Tab → X-Button neben queued-Zeile.
Workflows sind nicht Teil der Offline-Queue. Workflow-Steps gegen einen offline Agent laufen weiter in den klassischen Tactical-Timeout. Das ist Absicht, weil Workflows eigene Deadline-Semantik haben (step_deadline_at, force_after_hours).

Compliance

Required-Pakete sind Software die auf allen Clients installiert sein muss. Typisch: AV-Agent, Backup-Tool, interne Office-Suite.

  1. Paket als required markieren: Tab Pakete → ⚑-Icon klicken (aktiv = amber).
  2. Tab Compliance zeigt:
    • Wieviele Required-Pakete insgesamt
    • Wieviele Clients voll-konform (alle Required-Pakete vorhanden)
    • Pro Paket: Abdeckung (X/Y) und auf welchen Hostnames es fehlt
  3. Alles nachinstallieren: löst Install für jedes fehlende Required-Paket auf jedem noch-nicht-compliant Client aus.
Staged Required: Pakete die sowohl required als auch staged_rollout sind werden vom „Alles nachinstallieren" übersprungen (mit stage=all) — diese müssen manuell über den Rollouts-Tab ausgerollt werden.

Wie Installationen ablaufen

  1. Pre-Flight-Check: Ist der Agent in Tactical online? Wenn nicht → sofort Fehler.
  2. Bootstrap: Ein ~250-Byte PowerShell-Command wird via Tactical an den Agent gesendet. Dieser lädt das eigentliche Install-Script per HTTP vom Proxy und startet es im Hintergrund.
  3. Script-Ausführung: Das Script läuft auf dem Agent (Download, Install, Post-Verify). Nach jedem Schritt wird ein Progress-Callback an den Proxy gesendet — der Output erscheint live im Detail-Modal.
  4. Finaler Callback: Am Ende sendet das Script Exit-Code + Output. Der Proxy erkennt Soft-Errors, aktualisiert die DB und startet Re-Scans.

Auto-Retry

  • Choco .install-Variante: Wenn ein Choco-Uninstall mit „is not installed“ scheitert, wird automatisch die .install-Variante versucht (behebt Metapaket-Orphans wie vlc vs vlc.install).
  • Winget per-user-Fallback: Wenn --scope machine mit „No applicable installer“ scheitert und Scope auf auto steht, wird automatisch im User-Kontext wiederholt.

Timeout & Reboot-Check

  • Install-Timeout ist pro Paket konfigurierbar (Edit-Panel → „Install-Timeout“). Standard: 120s. Grosse Pakete (Office, SAP): 600–900s.
  • Reboot-Check: Wenn aktiviert, prüft das Script vor der Installation ob ein Neustart aussteht. Falls ja → Status Übersprungen statt Fehler.

Manueller Retry

Bei Fehler-Status: Klick auf ↻ (Retry-Button) in der Aktions-Zeile oder im Detail-Modal. Erstellt eine neue Aktion mit gleichen Parametern.

Vormerk-Abbruch

Wartungsfensters (Geplant)

  3. Zeitpunkt (deine lokale Zeitzone, datetime-local-Input).

Token widerrufen

Token widerrufen erhöht die token_version des Agents in der DB. Der alte Machine-JWT verliert sofort die Gültigkeit (Auth-Check schlägt fehl). Der Tray zeigt dem User einen Vollbild-„Token widerrufen"-Overlay mit „Neu installieren"-Button — bei Klick wird die Reinstall-URL aus den Settings geöffnet (falls gesetzt) oder der User wird auf IT verwiesen.

Anwendung: Agent wurde umgemeldet (neuer Besitzer), Hardware ausgetauscht ohne Setup-Re-Run, oder Verdacht auf Token-Kompromittierung.

Sperren / Entsperren

Sperren-Modal: Grund eingeben (Pflichtfeld). Ban-Eintrag landet in agent_blocklist — Pre-Flight-Check beim nächsten Dispatch lehnt mit „Agent gesperrt: <Grund>" ab. Tray-Polling läuft weiter aber alle Install-Anfragen scheitern. Sperrgrund + Datum erscheinen in der Agent-Detail-Header.

Entsperren: Entsperren in der Agent-Detail-Sicht. Löscht den Blocklist-Eintrag, Audit-Log erfasst den entsperrenden Admin.

Löschen

Löschen entfernt den Agent komplett (DB + zugehörige action_log/install_log via Cascade). Auf der Maschine bleibt der Tray installiert — beim nächsten Heartbeat bekommt er 401 (Token unbekannt) und zeigt den „Token widerrufen"-Overlay. Sinnvoll für ausgemusterte Geräte.

Tactical-Queue (Diagnose)

Endpoint GET /admin/api/tactical-queue liefert die Liste der gerade aktiven + wartenden Tactical-RMM-Kommandos pro Agent. Wird im Aktionen-Tab oben rechts als kleines „Tactical: N aktiv"-Indikator gerendert. Nützlich wenn Installations-Calls verzögert wirken — wenn die Queue voll ist, drosselt der Tactical-Worker.

Tray-Self-Update & Bulk-Push

Das Tray-EXE updatet sich nicht automatisch — der Push muss aktiv getriggert werden. Drei Wege.

Einzelne Maschine

  1. Tab Clients → Zeile → Update-Icon (rechts neben Agent-ID).
  2. Bei Online-Agent: Setup.exe wird via Tactical run_command auf den Agent geladen, Tray gekillt, Setup ausgeführt, neuer Tray gestartet. Dauert 1–3 Min.
  3. Bei Offline-Agent: Status Vorgemerkt — beim nächsten Online-Status wird automatisch dispatched (siehe Offline-Queue).

Bulk: Alle veralteten Clients

  1. Tab Clients → Banner mit Version-Verteilung („X aktuell · Y veraltet · Soll-Version: 2.6.0").
  2. Banner-Button Updates pushen.
  3. Bestätigen — alle Clients unter Soll-Version bekommen den Update-Befehl. Offline-Agents werden automatisch gequeued.

Bulk: Ausgewählte Clients (force-reinstall)

  1. Tab Clients → Checkboxen markieren.
  2. Bulk-Bar → Setup pushen.
  3. Im Modal: Force-Reinstall wenn auch Clients auf aktueller Version re-installiert werden sollen (Tray hängt, Registry-Cleanup erforderlich, o.ä.).

Soll-Version (Minimum-Required)

Settings → client_min_required_version setzen. Wenn ein Tray mit BUILD_VERSION < min_required pollt, zeigt der Tray einen nicht-wegklickbaren Force-Update-Modal. Sinnvoll wenn ein Security-Fix oder Pflichtfeature im Tray ausgerollt werden muss.

Leer lassen = kein Zwang. Der Tray prüft sonst lediglich „latest" vs „current" und blendet eine wegklickbare Banner-Notification ein.

Admin-Einladungen (v2.5.0)

Statt manuell Admin-User mit Passwort anzulegen, lädt man sie per Link ein — der Empfänger setzt selbst Benutzername + Passwort. Token-basiert, einmalig, mit Ablauf.

Einladung erstellen

  1. Tab Benutzer (Sidebar-Gruppe „Verwaltung", nur admin-Rolle).
  2. Abschnitt Einladungen → + Einladung.
  3. Im Modal: Rolle (admin/operator/viewer), Hinweis (optional, dem Eingeladenen angezeigt — z.B. „Für Marketing-Team-Onboarding"), Gültigkeit (1 / 7 / 14 / 30 Tage).
  4. Erstellen → Modal mit Link https://<dein-portal>/invite/<token>.
  5. Link kopieren und per sicherem Kanal (Signal, Teams-DM, etc.) an den Empfänger schicken.

Annahme-Flow (User-Sicht)

Empfänger öffnet den Link → Public-Seite (kein Login). Zeigt:

  • Einlader-Name + erstellt-Datum + Gültig-bis.
  • Die zugewiesene Rolle (kann nicht geändert werden).
  • Hinweis-Text wenn gesetzt.
  • Formular: Benutzername (frei wählbar, muss eindeutig sein), Anzeigename, Passwort (min. 8 Zeichen) + Wiederholung.

Submit → User wird mit der eingeladenen Rolle angelegt. Einladung wird atomar als „akzeptiert" markiert (kein Re-Use möglich). Empfänger ist automatisch eingeloggt.

Einladungen verwalten

Liste aller Einladungen im Benutzer-Tab mit Status (offen / akzeptiert / abgelaufen / widerrufen) und Aktionen:

  • Link kopieren — nur bei offenen Einladungen.
  • Widerrufen — Token wird sofort ungültig. Wenn der Empfänger den Link gerade öffnet, sieht er „Einladung widerrufen".
  • Löschen — Eintrag aus der DB entfernen (auch akzeptierte). Der erstellte User bleibt natürlich erhalten.
Kein SSO via Einladung. Einladungen legen lokale Accounts an. Microsoft-Entra-User werden separat über das SSO-Flow gemappt (siehe Settings → SSO). Ein Email-Hint hilft beim späteren Verknüpfen, ersetzt aber das SSO-Setup nicht.

Rate-Limit

Annahme-Endpoint POST /invite/{token} ist auf 10 Versuche pro 60s pro Client-IP begrenzt — Brute-Force-Schutz gegen Token-Raten. Bei zu vielen Versuchen kommt 429.

Deployment & Updates

Kiosk-Client auf PCs ausrollen

Drei Wege:

1. Tactical RMM Deploy-Script (empfohlen)

Tab Einstellungen → Abschnitt „Tactical-Deploy" → Snippet ist vorausgefüllt mit aktuellem Registration-Secret. In Tactical → Scripts → „PowerShell" → ins Script-Feld einfügen → auf Agent ausführen.

2. Manuelle Installation

{{SF_SLUG}}-setup.exe vom Proxy herunterladen ({{SF_SETUP_EXE_URL}}) und als Admin ausführen. Der Installer registriert den Agent mit dem im EXE einkompilierten PROXY_URL und verwendet das in der Setup-GUI eingegebene Registration-Secret.

3. Per GPO / SCCM

Gleiche EXE mit /S /RegSecret="xxx" Silent-Flags.

Nach Installation registriert sich der Agent automatisch beim Proxy und erscheint im Clients-Tab.

Client-EXEs bauen

Wenn du Proxy-URL, Branding oder Icon geändert hast:

  1. Tab Einstellungen → Abschnitt „Client-EXEs" → EXEs bauen.
  2. Build-Progress wird live angezeigt. Dauert ~3-4 Minuten (Wine + PyInstaller).
  3. Nach Abschluss sind die neuen EXEs unter /download/ verfügbar. Das System markiert den neuen Build automatisch als is_current — Tray-Self-Update verwendet ab sofort diese Version.

Version-Override

Standardmäßig wird die Version aus client/_tray_version.py gelesen. Im Build-Modal optional Version-Override setzen (z.B. 2.6.1-rc1 für einen Release-Candidate ohne Code-Bump). Override muss [a-zA-Z0-9][a-zA-Z0-9._\-+]* matchen.

Direktes Triggern des Builders (z.B. via docker exec + urllib) überspringt die finish_build_log-Markierung — der Build wandert nicht als is_current in die DB. Immer über die Admin-UI / POST /admin/api/build triggern.

Build-History

Settings → Abschnitt „Client-EXEs" → unter dem Build-Button die Liste aller bisherigen Builds chronologisch. Pro Build:

  • Version (z.B. 2.6.0), Datum, Trigger (Admin-User).
  • Status: running / success / failed.
  • SHA256 der beiden EXEs (Setup + Tray) — wird vom Tray für Integrity-Check beim Self-Update verwendet.
  • Log: Klick zeigt den vollen Build-Log (PyInstaller-Output, hilfreich bei failed-Builds).
  • Als aktuell setzen: Bei einem alten erfolgreichen Build → ihn wieder zum is_current machen. Sinnvoll wenn der neueste Build kaputt ist und du auf einen funktionierenden zurückrollen willst.

TLS / Reverse-Proxy

Der Proxy kann selbst kein TLS. Im Produktiv-Setup läuft Caddy oder Traefik vor dem Container und terminiert TLS. Wichtig: X-Forwarded-Proto: https muss vom Reverse-Proxy gesetzt werden, sonst landen Session-Cookies nicht als Secure und Browser blocken sie in HTTPS-Kontexten.

Wenn Caddy/Traefik nicht auf 127.0.0.1 läuft (z.B. separater Container), muss seine IP unter trusted_proxies eingetragen sein — sonst ignoriert der Rate-Limiter den XFF-Header und alle Requests gehen über den Proxy-Bucket.

SSO einrichten

  1. Microsoft Entra → App-Registrierung → Neue Registrierung. Name z.B. „Softshelf Admin".
  2. Redirect-URI: <proxy_public_url>/admin/sso/callback.
  3. API-Permissions: openid, email, profile, User.Read.
  4. Client-Secret erzeugen.
  5. In den Softshelf-Settings unter SSO eintragen: Tenant-ID, Client-ID, Client-Secret, erlaubte Email-Domains.
  6. Login-Seite zeigt jetzt einen „Mit Microsoft anmelden"-Button. Bei erstem Login: der eingeloggte Entra-User wird als operator-Account angelegt (Admin muss anschließend die Rolle hochsetzen).

Auf neuen Host migrieren

  1. Backup: SQLite-DB + Uploads. Siehe Backup & Restore.
  2. Auf neuem Host: git clone + docker-compose up -d.
  3. Backup in /app/data/ einspielen, Container neu starten.
  4. Wenn sich die Proxy-URL ändert: Einstellungen → Proxy Public URL aktualisieren → EXEs bauen.
  5. Alte Clients müssen neu deployed werden (Registration-Secret + evtl. neue Proxy-URL).

Backup & Restore

Minimal-Backup:

rsync -av /opt/softshelf/data/softshelf.db backup:/backups/
rsync -av /opt/softshelf/data/uploads/ backup:/backups/

Was nicht gesichert werden muss:

  • winget_index.db — wird nightly aus Microsoft-CDN regeneriert
  • downloads/*.exe — via „EXEs bauen" regenerierbar
  • Docker-Container-State — alles in der DB

Restore: DB zurückkopieren, Container restart. Secrets in .env müssen identisch sein (sonst Machine-Tokens invalid).

Sicherheit

Auth-Pfade

Kiosk-ClientJWT (HS256) mit agent_id, hostname, token_version. Secret aus .env. Revoke via token_version-Bump oder Sperre.
Admin-SessionHttpOnly-Cookie, SameSite=Lax, Secure wenn Reverse-Proxy TLS liefert. Passwort-Hash: scrypt.
SSOMicrosoft-Entra-OIDC mit email_verified-Check.

CSRF

Middleware akzeptiert POST/PATCH/DELETE nur mit matched Origin-Header oder X-Requested-With: XMLHttpRequest. GET ist frei (state-nondestructive).

Rate-Limiting

Per-IP Sliding-Window. X-Forwarded-For vom Loopback-Proxy vertraut.

Input-Validierung

  • Paketnamen per Regex (Typ-abhängig).
  • SQL immer parameterized — keine f-string-Interpolation.
  • Pydantic-Bodies mit Field-Validators (Max-Length, Regex, Whitelist-Sets).
  • Frontend: esc() + jsStr() gegen XSS.

Upload-Härtung

  • SHA256-Verifizierung.
  • Path-Traversal-Check bei ZIP-Extraktion.
  • MSI-Metadata via msiinfo (kein Python-Parser).
  • Icon-Upload: Pillow mit DecompressionBombError-Schutz.

Build-Pipeline

  • build.sh nimmt Parameter nur per Env-Vars (keine Shell-Injection).
  • PyInstaller läuft in Wine-Sandbox im separaten Container.

Operative Checkliste

  • Regelmäßig Registration-Secret rotieren (Einstellungen → Rotate-Button).
  • Token-TTL setzen (Default 365 Tage).
  • Log-Retention setzen (Default 90 Tage).
  • Inaktive Admin-Accounts deaktivieren (cascadet automatisch alle Sessions).
  • Reverse-Proxy mit TLS und X-Forwarded-* Headern (Caddy/Traefik).

SymptomUrsache / Fix
Kiosk-Client erscheint nach Deploy nicht im Tray Tactical läuft als SYSTEM (Session 0). Der Installer erzeugt einen Task-Scheduler-Task mit Principal=active user — check mit schtasks /Query /TN {{SF_SLUG}} im User-Context.
winget-Install scheitert mit „No applicable installer found" Exit-Code -1978335216. Bei scope=auto: Layer-2-Fallback tried automatically. Sonst: Scope auf user setzen.
choco-Install sagt „0/1 packages" Paket ist FOSS-broken oder 404. Im Fehler-Tab ist Details + Link zum Choco-Log auf dem Agent.
Rollout bleibt in Phase 1 stehen (1) Nicht „Weiter" geklickt. (2) Auto-Advance aus. (3) Auto-Advance pausiert wegen offener Fehler — Fehler-Tab prüfen + bestätigen.
Operator sieht disabled Buttons Kein Bug — RBAC. Admin-Rolle nötig für Whitelist-Edit, Ring-Mutation, Required-/Staged-Flags.
Viewer bekommt 403 bei jedem Klick Viewer-Rolle hat nur GETs. Keine Aktionen möglich.
Scheduled Job lief nicht zur eingestellten Zeit Tick läuft minuten-basiert, ±1 Min Toleranz. Bei Container-Restart innerhalb des Fensters werden Jobs beim nächsten Tick abgearbeitet.
Workflow-Run bleibt auf „running" stehen, kein Fortschritt Der Agent hat keinen Callback geschickt. Mögliche Ursachen: (1) Tactical run_command konnte den Agent nicht erreichen (Agent offline). (2) Script ist abgestürzt ohne Exit-Code. (3) Die Callback-URL ist vom Agent nicht erreichbar (Proxy-URL falsch konfiguriert). Run nach Diagnose manuell abbrechen und neu starten.
Reboot-Dialog erscheint nicht auf dem Client Der Kiosk-Client muss laufen und ein User muss eingeloggt sein. Prüfen: Tray-Icon vorhanden? Client pollt /api/v1/health alle 30s. Wenn kein Client läuft: der Zwangs-Neustart nach force_after_hours greift trotzdem.
Workflow startet nicht — „aktiver Run bereits vorhanden" Pro Agent ist maximal ein aktiver Run erlaubt. Den laufenden Run auf der Agent-Detail-Seite abbrechen, dann neu starten.
Version-Picker zeigt keine Versionen für Choco-Paket Die Versions-Liste kommt aus dem Fleet-Scan (agent_choco_state). Wenn das Paket noch auf keinem Agent installiert wurde, ist sie leer. Version manuell eintippen — das Feld akzeptiert freien Text.
