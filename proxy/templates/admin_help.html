Schnellstart

Die fünf häufigsten Aktionen:

📦

Paket freigeben

Tab Pakete → + Paket hinzufügen → winget/choco/custom wählen → Name suchen → +.

🚀

Staged Rollout starten

Tab Pakete → ▶-Icon bei staged Paket. Phase 1 (Canary) geht sofort los, danach im Rollouts-Tab weiter.

⚠

Fehler abarbeiten

Tab Fehler zeigt offene Install-Fehler flottenweit. Erneut oder ✓ zum Bestätigen.

✅

Required-Compliance prüfen

Tab Compliance: Required-Pakete mit Abdeckung pro Agent. „Alles nachinstallieren" für einen Rutsch.

⏰

Wartungsfenster anlegen

Tab Geplant → + Neuer Job → Zeitpunkt + Aktion + Stage wählen. Läuft automatisch zur eingestellten Zeit.

🔍

Quick-Jump

⌘K / Strg+K öffnet Blitzsuche über alle Tabs, Pakete, Agents und Profile.

Rollen & Rechte (RBAC)

Softshelf kennt drei Rollen. Eingestellt pro Admin-User unter Benutzer.

RolleDarfDarf nicht
admin Alles: Whitelist, Benutzer, Einstellungen, Build, Dispatch, Paket-Flags, Agent-Ringe —
operator Dispatch (Install/Uninstall/Update-All), Push-Update, Bulk-Aktionen, Rollouts starten & weiterschalten, Profile anwenden, Fehler bestätigen Whitelist-Edit, Users anlegen, Settings ändern, Build triggern, Required/Staged-Flags setzen, Ring umkonfigurieren
viewer Alle GETs (lesen, durchklicken, CSV-Export) Jegliche state-changing Action → 403
Durchsetzung: Der Server ist Source of Truth. Die UI versteckt oder disabled Admin-Only Controls für andere Rollen kosmetisch, aber jeder API-Call wird backend-seitig in _require_admin gegen method+path geprüft.

Paket-Typen

Softshelf kennt drei Paket-Quellen. Type-Pills in der UI zeigen den Typ mit farbigem Dot:

TypQuelleDispatchBesonderheit
winget Microsoft-winget-Catalog, lokal gemirrowt winget install/upgrade/uninstall via Tactical run_command Scope (auto/machine/user) pro Paket — Layer-2-Fallback für per-user-only-Installer wie LastPass, Bitwarden.
choco Chocolatey Community choco install/uninstall -y via run_command mit Output-Capture Auto-Ghost-Repair: --force + --skip-autouninstaller bei unsauberen Uninstalls.
custom Manueller MSI/EXE/Ordner-Upload Download via signed-URL → msiexec/Start-Process Version-Tracking, Push-Updates.

Winget-Paket freigeben

  1. Tab Pakete → + Paket hinzufügen.
  2. Reiter Winget. Tippen → Live-Suche im lokalen Catalog-Mirror.
  3. Plus-Button bei Treffer → aktiviert das Paket in der Whitelist.
  4. Optional: Scope inline einstellen:
    • auto (Default): erst SYSTEM + --scope machine, bei „No applicable installer" auto-retry im User-Kontext.
    • machine: nur SYSTEM, kein Fallback.
    • user: direkt User-Kontext — für LastPass, Bitwarden, Firefox-per-user.
Discovery: Das Panel zeigt außerdem „In der Flotte gefunden" — winget-Pakete die auf euren Clients bereits installiert sind aber noch nicht in der Whitelist. One-click-Aktivieren möglich.

Choco-Paket freigeben

  1. Tab Pakete → + Paket hinzufügen → Reiter Aus Chocolatey.
  2. Name tippen, Plus drücken.
FOSS-Hinweis: Manche choco-Pakete (die mit .install-Suffix) laden Installer vom Hersteller-CDN, nicht von Chocolatey selbst. Bei 404 oder „Likely broken for FOSS users" steht die Fehlermeldung mit Link zum Paket-Log im Fehler-Tab.

Eigenes MSI/EXE/Ordner hochladen

  1. Tab Pakete → + Paket hinzufügen → Eigene Datei / Programm-Ordner.
  2. Datei oder Ordner wählen. MSI-Metadaten werden automatisch extrahiert.
  3. Optional: Silent-Install-Args anpassen. Default für MSI: /qn /norestart.
  4. Uninstall-Command: via Detect-Button aus Tactical übernehmen oder manuell hinterlegen.
  5. Hochladen.

Nachträgliches Bearbeiten: Stift-Icon in der Paket-Zeile → Panel mit allen Feldern, inklusive Versions-History und „Inhalt verwalten" bei Ordner-Paketen.

Bulk-Import winget-IDs

Wenn du 50 Pakete auf einmal aktivieren willst:

  1. Tab Pakete → Bulk-Import.
  2. Textarea: eine winget-ID pro Zeile (oder komma-getrennt).
  3. Kategorie setzen.
  4. Importieren. Metadaten (Name, Publisher) werden automatisch aus dem Catalog geholt. Bereits aktive Pakete werden übersprungen, ungültige IDs landen in der Fehler-Liste.

Profile

Profile sind Bundles aus mehreren Paketen, zuweisbar auf beliebige Clients.

  1. Tab Profile → + Neues Profil → Name, Beschreibung, Farbe.
  2. Paket-Picker: welche Pakete enthält das Profil. Optional: Version-Pin pro Paket.
  3. Speichern → Profil erscheint in der Liste links.
  4. Client zuweisen: Profil auswählen → Clients zuweisen → Agent-Picker.
  5. Apply löst sofort Install für alle nicht-vorhandenen Pakete aus.
Auto-Update: Option im Profil-Editor. Wenn gesetzt, pusht der nightly Job (03:00 UTC) Updates für alle Pakete im Profil an die zugewiesenen Agents. Smart-Skip: bereits aktuelle Installs werden übersprungen.
Propagation: Fügst du ein Paket zu einem Profil hinzu, wird es sofort auf allen zugewiesenen Clients installiert (außer sie haben's schon). Entfernen ohne Uninstall — der Admin muss explizit „Unassign mit Uninstall" wählen um zu deinstallieren.

Phased Rollouts

Konzept: Ringe, Phasen und Zuordnung

Zwei Dinge klar trennen

Ein Staged Rollout beantwortet nur eine Frage: wann bekommt ein Client das Update? Ob ein Paket überhaupt zu einem Client gehört, entscheidet eine andere Mechanik — die Zuordnung.

MechanikAntwortet aufWird gesteuert durch
Zuordnung Ist dieses Paket auf diesem Client installiert (oder sollte es sein)? Profil-Zuweisung · Required-Flag · direkter Admin-Push · User-Install im Kiosk
Rollout-Phase Wann geht das Update zu welchem Ring? Staged-Flag + Ring pro Agent + aktive Rollout-Phase
Häufiger Irrtum: Staged-Flag aktivieren reicht nicht aus damit ein Paket auf neue Clients installiert wird. Staged steuert nur den Rollout bestehender Zuweisungen. Ein Client der das Paket gar nicht hat, bekommt es auch bei erreichter Phase nicht — egal wie weit der Rollout ist.

Konkrete Beispiele

AusgangslageAdmin-AktionErgebnis auf dem Client
Firefox installiert auf Client A (Ring 1). Staged aktiv, Rollout Phase 1 Neue Version 128 verfügbar ✓ Client A bekommt Update (Ring 1 ≤ Phase 1)
Firefox installiert auf Client B (Ring 3). Rollout Phase 1 — ✗ Kein Update-Button im Kiosk (Ring 3 > Phase 1). Wartet auf Phase 3
Firefox nicht auf Client C installiert. Staged + Phase 3 — ✗ Firefox erscheint nicht auf Client C — fehlende Zuordnung. Rollout ignoriert ihn komplett
Firefox in Profil „Devs" enthalten, Client C hat das Profil. Staged + Phase 1 Profil-Apply bei Zuweisung Firefox wird auf Client C installiert (Profil-Apply). Updates respektieren dann Ring/Phase

Ringe konfigurieren

Jeder Agent hat genau einen Ring: Ring 1 (Canary), Ring 2 (Pilot), oder Ring 3 (Produktion = Default für neu registrierte Agents). Setzbar im Agent-Detail-Header (admin-only). Labels pro Umgebung umbenennbar unter Rollouts → Einstellungen.

Phasen-Mapping

Eine Rollout-Phase löst an einen Ring aus:

  • Phase 1 → alle Agents mit Ring 1
  • Phase 2 → alle Agents mit Ring 2
  • Phase 3 → alle Agents mit Ring 3 (Produktion)

Durch Kiosk-Gate sehen Agents in späteren Ringen das Update erst wenn ihre Phase erreicht ist — auch bei User-initiierten Update-Checks im Kiosk.

Pakete mit staged_rollout=1 (Haken-Icon in Pakete-Row) müssen diese Phasen durchlaufen:

Phase 1
Ring 1 (Canary)
kleinste Testgruppe
→
Phase 2
Ring 2 (Pilot)
größere Testgruppe
→
Phase 3
Produktion
alle Nicht-Test-Agents

Zwischen den Phasen prüft der Admin (oder der Auto-Modus) ob Fehler auftraten. Bei Problemen: Rollout abbrechen oder pausieren.

Workflow: Paket staged ausrollen

  1. Paket als phased markieren: Tab Pakete → ✓-Icon pro Zeile klicken (aktiv = amber).
  2. Agent-Ringe zuweisen: Tab Clients → Agent öffnen → Ring-Dropdown im Header → 1 für Test-Gerät setzen.
  3. Rollout starten: Tab Pakete → ▶-Icon bei phased Paket mit Updates. Bestätigen → Phase 1 wird sofort ausgelöst.
  4. Status prüfen: Tab Rollouts zeigt aktive Rollouts. Oder auf Übersicht.
  5. Fehler checken: Tab Fehler nach Problemen durchsehen.
  6. Weiter: Weiter-Button im Rollouts-Tab → Phase 2 (Ring 2).
  7. Nochmal prüfen, dann Weiter → Phase 3 (Produktion).
  8. Nach Phase 3 ist Rollout automatisch done.
Abbruch: Abbrechen-Button stoppt den Rollout. Bereits dispatcht Phasen bleiben auf den Agents — Rollback ist ein separater Uninstall (Profil-Unassign mit Uninstall oder manuelles choco/winget uninstall).

Auto-Advance-Einstellungen

Tab Rollouts → Abschnitt „Einstellungen":

Ring-LabelsAnzeigenamen für Ring 1-3 (z.B. „QA-Team", „Pilot-Gruppe").
Neue winget-Pakete automatisch stagedPolicy: bei Bulk-Import oder Aktivierung wird staged_rollout=1 gesetzt.
Auto-Advance aktivierenScheduler-Tick alle 15 Min: schaltet aktive Rollouts nach Ablauf der Wartezeit in die nächste Phase.
Wartezeit (Std)Minimum zwischen Phasen für Auto-Advance. Default 24h.
Max-Fehler-Rate (%)Wenn mehr als X% der Agents in der aktuellen Phase Fehler haben: pausieren statt weiterschalten. 0 = schon ein Fehler blockt.

Compliance

Required-Pakete sind Software die auf allen Clients installiert sein muss. Typisch: AV-Agent, Backup-Tool, interne Office-Suite.

  1. Paket als required markieren: Tab Pakete → ⚑-Icon klicken (aktiv = amber).
  2. Tab Compliance zeigt:
    • Wieviele Required-Pakete insgesamt
    • Wieviele Clients voll-konform (alle Required-Pakete vorhanden)
    • Pro Paket: Abdeckung (X/Y) und auf welchen Hostnames es fehlt
  3. Alles nachinstallieren: löst Install für jedes fehlende Required-Paket auf jedem noch-nicht-compliant Client aus.
Staged Required: Pakete die sowohl required als auch staged_rollout sind werden vom „Alles nachinstallieren" übersprungen (mit stage=all) — diese müssen manuell über den Rollouts-Tab ausgerollt werden.

Verteilung & Updates

Tab Verteilung zeigt pro Paket: wer hat's, wer ist outdated.

  • Suche: Paketnamen filtern.
  • Type-Pills: nur winget / choco / custom.
  • Nur mit Updates: Checkbox blendet aktuelle Pakete aus.
  • Spalten-Header sortierbar (Veraltet desc = Default).
  • Zeilen-Klick: Slide-Over rechts mit Agent-Liste für das Paket.
  • Checkbox-Spalte: Multi-Select → Aktions-Leiste oben („Updates verteilen" / „Überall deinstallieren").
  • Update-Button pro Zeile: Flotten-Update nur für dieses Paket (mit Ring-Auswahl).
Stage-Picker: Bei jedem Dispatch erscheint ein Modal mit Radio-Buttons für Ring 1 / Ring 2 / Ring 3 / Alle Test-Ringe / Nur Produktion / Alle Agents. Bei staged_rollout-Paketen ist „Alle Agents" deaktiviert.

Fehler & Acknowledgement

Tab Fehler ist die Single-Source-of-Truth für alle offenen Install-Fehler flottenweit.

Spalten

Aktioninstall/upgrade/uninstall

ZeitWann

Bestätigen vs. Erneut

  • Erneut startet den letzten Versuch neu — sinnvoll bei transienten Fehlern (Netzwerk, Tactical kurz weg).
  • ✓ markiert als bestätigt. Fehler verschwindet aus der Liste, aber Fehler-Text und voller Output bleiben in der DB. „Bestätigte anzeigen"-Toggle zeigt sie wieder.
  • Alle bestätigen als Bulk-Aktion.
  • Wenn ein neuer Fehler zu demselben Agent kommt, wird das Ack automatisch resetet.
Agent-Detail-Banner: Solange ein Fehler nicht acked ist, erscheint er als rotes Banner oben im Agent-Detail mit Voller Output und Bestätigen Buttons.

Aktionen-Log

Tab Aktionen zeigt alle Install-, Uninstall- und Upgrade-Aktionen chronologisch mit Live-Fortschritt.

Status-Lifecycle

AusstehendAktion angelegt, Bootstrap noch nicht gesendet.
LäuftBootstrap gesendet, Script läuft auf dem Agent. Progressive Output-Updates erscheinen live.
ErfolgCallback empfangen, Exit-Code OK, keine Soft-Errors.
FehlerCallback empfangen mit Fehler, oder Delivery fehlgeschlagen, oder Timeout.
ÜbersprungenPre-Check hat Installation verhindert (z.B. Neustart steht aus).

Wie Installationen ablaufen

  1. Pre-Flight-Check: Ist der Agent in Tactical online? Wenn nicht → sofort Fehler.
  2. Bootstrap: Ein ~250-Byte PowerShell-Command wird via Tactical an den Agent gesendet. Dieser lädt das eigentliche Install-Script per HTTP vom Proxy und startet es im Hintergrund.
  3. Script-Ausführung: Das Script läuft auf dem Agent (Download, Install, Post-Verify). Nach jedem Schritt wird ein Progress-Callback an den Proxy gesendet — der Output erscheint live im Detail-Modal.
  4. Finaler Callback: Am Ende sendet das Script Exit-Code + Output. Der Proxy erkennt Soft-Errors, aktualisiert die DB und startet Re-Scans.

Auto-Retry

  • Choco .install-Variante: Wenn ein Choco-Uninstall mit „is not installed“ scheitert, wird automatisch die .install-Variante versucht (behebt Metapaket-Orphans wie vlc vs vlc.install).
  • Winget per-user-Fallback: Wenn --scope machine mit „No applicable installer“ scheitert und Scope auf auto steht, wird automatisch im User-Kontext wiederholt.

Timeout & Reboot-Check

  • Install-Timeout ist pro Paket konfigurierbar (Edit-Panel → „Install-Timeout“). Standard: 120s. Grosse Pakete (Office, SAP): 600–900s.
  • Reboot-Check: Wenn aktiviert, prüft das Script vor der Installation ob ein Neustart aussteht. Falls ja → Status Übersprungen statt Fehler.

Manueller Retry

Bei Fehler-Status: Klick auf ↻ (Retry-Button) in der Aktions-Zeile oder im Detail-Modal. Erstellt eine neue Aktion mit gleichen Parametern.

Detail-Modal: Klick auf eine Zeile im Aktionen-Tab öffnet ein Modal mit: Agent, Paket, Typ, Exit-Code, Dauer, Fehlermeldung und aufklappbarem vollem PowerShell-Output.

Wartungsfensters (Geplant)

Admin legt zeitgesteuerte Dispatches an. APScheduler-Tick alle Minute prüft Pending-Jobs.

  1. Tab Geplant → + Neuer Job.
  2. Beschreibung (optional).
  3. Zeitpunkt (deine lokale Zeitzone, datetime-local-Input).
  4. Aktion wählen:
    • Updates pushen für ein Paket — einzeln
    • Bulk: Updates für mehrere Pakete verteilen — Multi-Select
    • Required-Pakete nachinstallieren — Compliance-Fix
  5. Stage wählen (Ring 1 / Ring 2 / Prod / Alle).
  6. Einplanen.

Der Tick führt den Job zur ersten vollen Minute ≥ run_at aus und markiert ihn als done oder failed mit Ergebnis.

Genauigkeit: Tick ist minuten-basiert, also ±1 Min. Für echte Sekunden-Genauigkeit müsste der Scheduler anders gebaut sein. Für Wartungsfensters ist das ausreichend.

Deployment & Updates

Kiosk-Client auf PCs ausrollen

Drei Wege:

1. Tactical RMM Deploy-Script (empfohlen)

Tab Einstellungen → Abschnitt „Tactical-Deploy" → Snippet ist vorausgefüllt mit aktuellem Registration-Secret. In Tactical → Scripts → „PowerShell" → ins Script-Feld einfügen → auf Agent ausführen.

2. Manuelle Installation

softshelf-setup.exe vom Proxy herunterladen (http://proxy:8765/download/softshelf-setup.exe) und als Admin ausführen. Der Installer registriert den Agent mit dem im EXE einkompilierten PROXY_URL und verwendet das in der Setup-GUI eingegebene Registration-Secret.

3. Per GPO / SCCM

Gleiche EXE mit /S /RegSecret="xxx" Silent-Flags.

Nach Installation registriert sich der Agent automatisch beim Proxy und erscheint im Clients-Tab.

Client-EXEs bauen

Wenn du Proxy-URL, Branding oder Icon geändert hast:

  1. Tab Einstellungen → Abschnitt „Client-EXEs" → EXEs bauen.
  2. Build-Progress wird live angezeigt. Dauert ~3-4 Minuten (Wine + PyInstaller).
  3. Nach Abschluss sind die neuen EXEs unter /download/ verfügbar.

Auf neuen Host migrieren

  1. Backup: SQLite-DB + Uploads. Siehe Backup & Restore.
  2. Auf neuem Host: git clone + docker-compose up -d.
  3. Backup in /app/data/ einspielen, Container neu starten.
  4. Wenn sich die Proxy-URL ändert: Einstellungen → Proxy Public URL aktualisieren → EXEs bauen.
  5. Alte Clients müssen neu deployed werden (Registration-Secret + evtl. neue Proxy-URL).

Backup & Restore

Minimal-Backup:

rsync -av /opt/softshelf/data/softshelf.db backup:/backups/
rsync -av /opt/softshelf/data/uploads/ backup:/backups/

Was nicht gesichert werden muss:

  • winget_index.db — wird nightly aus Microsoft-CDN regeneriert
  • downloads/*.exe — via „EXEs bauen" regenerierbar
  • Docker-Container-State — alles in der DB

Restore: DB zurückkopieren, Container restart. Secrets in .env müssen identisch sein (sonst Machine-Tokens invalid).

Sicherheit

Auth-Pfade

Kiosk-ClientJWT (HS256) mit agent_id, hostname, token_version. Secret aus .env. Revoke via token_version-Bump oder Sperre.
Admin-SessionHttpOnly-Cookie, SameSite=Lax, Secure wenn Reverse-Proxy TLS liefert. Passwort-Hash: scrypt.
SSOMicrosoft-Entra-OIDC mit email_verified-Check.

CSRF

Middleware akzeptiert POST/PATCH/DELETE nur mit matched Origin-Header oder X-Requested-With: XMLHttpRequest. GET ist frei (state-nondestructive).

Rate-Limiting

Per-IP Sliding-Window. X-Forwarded-For vom Loopback-Proxy vertraut.

Input-Validierung

  • Paketnamen per Regex (Typ-abhängig).
  • SQL immer parameterized — keine f-string-Interpolation.
  • Pydantic-Bodies mit Field-Validators (Max-Length, Regex, Whitelist-Sets).
  • Frontend: esc() + jsStr() gegen XSS.

Upload-Härtung

  • SHA256-Verifizierung.
  • Path-Traversal-Check bei ZIP-Extraktion.
  • MSI-Metadata via msiinfo (kein Python-Parser).
  • Icon-Upload: Pillow mit DecompressionBombError-Schutz.

Build-Pipeline

  • build.sh nimmt Parameter nur per Env-Vars (keine Shell-Injection).
  • PyInstaller läuft in Wine-Sandbox im separaten Container.

Operative Checkliste

  • Regelmäßig Registration-Secret rotieren (Einstellungen → Rotate-Button).
  • Token-TTL setzen (Default 365 Tage).
  • Log-Retention setzen (Default 90 Tage).
  • Inaktive Admin-Accounts deaktivieren (cascadet automatisch alle Sessions).
  • Reverse-Proxy mit TLS und X-Forwarded-* Headern (Caddy/Traefik).
